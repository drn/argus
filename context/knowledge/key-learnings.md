## Key Learnings

- **`ctrl+c` only exits Argus from the root (task list) view.** The global `handleGlobalKey` processes `KeyCtrlC` before mode-specific handlers. In agent mode, `ctrl+c` is always consumed: if the session is alive, `0x03` is written to the PTY; if dead, it's a no-op. Only in non-agent modes does `ctrl+c` call `tapp.Stop()`. This prevents accidentally killing Argus when trying to interrupt an agent process.

- **`ctrl+q` in diff mode must both exit diff mode AND refocus the agent panel.** When a file diff is open, pressing `ctrl+q` calls `exitDiffMode()` but must also refocus the terminal panel — without this, the user lands back on the files panel and needs a second keypress to reach the terminal. The global `ctrl+q` handler runs before the diff key handler, so the fix belongs there.

- **Opening a PR from agent view: `ctrl+p` when session active, `o` when finished.** Use `ctrl+p` (raw byte 0x10) to open the PR URL in the browser — works in every terminal without configuration. Avoid modifier+key shortcuts (Cmd+O, Option+O) — macOS terminals intercept ⌘+key at the app level, and Option+O sends `ø` (U+00F8) unless the terminal is configured with "Use Option as Meta key" which most users don't have enabled. `ctrl+p` is intercepted BEFORE the key reaches the PTY, so it works while the agent is running. Plain `o` is also intercepted when the session is finished (PTY is dead). The PR URL is stored on the agent view and kept current by: (1) DB lookup at view entry; (2) live PR URL detection during agent run; (3) final output scan on agent finish.

- **New task form has `/skill` (or `$skill` for codex) autocomplete with project-specific skills.** `internal/skills/` scans `~/.claude/skills/` AND the selected project's `<projectPath>/.claude/skills/` for skill directories. Project-specific skills take precedence on name collision (extraDirs are searched first). The trigger character is `/` for claude backends and `$` for codex backends — determined by checking `agent.IsCodexBackend()` on the selected backend's command. When the prompt value starts with the trigger and contains no spaces, a filtered dropdown appears. Changing the project selection reloads skills asynchronously.

- **New task backend selector pre-selects the global default backend — no inherit option.** The selector shows only the sorted backend names (no `(inherit)` entry). The default backend is pre-selected by name at open, falling back to index 0. `SelectedBackend()` always returns an explicit name; `Task().Backend` is never empty.

- PTY child processes need a real terminal size at launch (`pty.StartWithSize`), not 0x0. TUI apps like claude won't render with zero dimensions.
- Use `charmbracelet/x/term` for raw mode (cross-platform) instead of platform-specific ioctls (`TIOCGETA` is macOS-only, `TCGETS` is Linux-only).
- The single-reader-tee pattern (one goroutine reads PTY, tees to buffer + optional writer) is critical. Two goroutines reading the same fd causes data loss.
- **Never run git commands or filesystem discovery synchronously on the UI thread.** Even fast git commands take 50-500ms which freezes the entire TUI. Run them in background goroutines and deliver results via callbacks or messages. Cache resolved paths to avoid repeated lookups.
- **Backend default config must be self-healing.** The default claude backend command MUST include `--dangerously-skip-permissions`, MUST NOT use `-p` as the prompt flag, and MUST NOT include `--worktree` (Argus manages worktrees itself via `git worktree add`). `-p` puts Claude in non-interactive print mode (process prompt → exit), which silently breaks agent sessions. The `fixupBackends()` method in `internal/db/migrate.go` runs on every `Open()` to detect and repair outdated backend configs. Any change to `DefaultConfig()` backend values must be mirrored in `fixupBackends()` detection logic, or existing databases will retain stale values.
- **`removeWorktree` must validate paths before `os.RemoveAll`.** The `os.RemoveAll` fallback in `removeWorktree` will nuke any directory if `git worktree remove` fails — including the root project if `t.Worktree` is incorrectly set. The `isWorktreeSubdir` guard ensures the path contains `/.argus/worktrees/` or `/.claude/worktrees/` before any removal. Any new cleanup code that calls `os.RemoveAll` on user-provided paths must have a similar safety check.
- **Use `ansi.Hardwrap` not `ansi.Truncate` for panel content that should wrap.** The `charmbracelet/x/ansi` package provides both: `Truncate` clips lines at a width (good for single-line labels), while `Hardwrap` inserts newlines at the width boundary preserving ANSI escape sequences (good for multi-line content panels like diff views). When using `Hardwrap`, pre-compute and cache the wrapped lines since wrapping is O(content_size) and the width only changes on resize. Invalidate the cache when content changes or width changes. Scrolling must operate on visual (wrapped) lines, not source lines.
- **Always render the cursor regardless of `CursorVisible()`.** TUI agents like Claude Code (built with Ink) hide the hardware cursor via `\x1b[?25l` — standard practice for TUI apps that manage their own rendering. The x/vt emulator correctly tracks this, so `CursorVisible()` returns `false`. The cursor must be rendered unconditionally — `CursorVisible()` is irrelevant for Argus's use case since the agent view always wants to show the cursor position.
- **Worktree creation must succeed before task persistence.** Never `db.Add(task)` without a valid worktree. If `CreateWorktree` fails, keep the new task form open with the error — do NOT fall back to running in the project directory or delegating to `--worktree`. `CreateWorktree` returns `(wtPath, finalName, err)` — the `finalName` may differ from the input if name conflicts required a `-1`, `-2` suffix.
- **Map lookups returning `*T` become non-nil interfaces.** When a method returns an interface (e.g., `Get(id) SessionHandle`) and the underlying implementation uses a map (`sessions[id]`), a missing key returns `nil *Session`. Assigning this to an interface gives a **non-nil interface** with a nil concrete value — `== nil` checks fail. The `Get` method must explicitly check `if sess == nil { return nil }` before returning. This applies to any method on Runner/Client that returns `SessionHandle`.
- **Stream loss ≠ process exit — use `StreamLost` flag to distinguish.** When a stream connection fails, `connectStream` routes to one of three paths: (1) **process exited**: call `removeSession` (queries `GetExitInfo`, fires exit notification → normal exit flow); (2) **stream lost / process alive**: fire exit notification with `StreamLost: true` → TUI keeps task `InProgress` and shows a warning; (3) **daemon unreachable**: also treat as stream lost — can't confirm process exit, so keep `InProgress`. `isSessionAlive()` returns `(alive bool, reachable bool)` — when `reachable=false` (RPC failed), treat as stream lost, not process exit. Retry exhaustion (3 retries × 500ms) is also stream lost, not process exit.
- **`client.Get()` must return `nil` for any non-alive session.** The original condition `!info.Alive && info.PID == 0` is too lenient. During the window between `onFinish()` firing and `delete(r.sessions, taskID)` executing in the runner, `SessionStatus` returns `{Alive: false, PID: non-zero}` (process has exited but the session hasn't been removed yet). The lenient check lets `Get()` create a new `RemoteSession` with its own `connectStream` goroutine, firing a second exit notification for the same task. Fix: use `!info.Alive` alone — the PID is a stale artifact. A dead session must never be returned as a valid handle regardless of PID.
- **Daemon health check via failure counter on tick.** The tick handler pings the daemon via `client.Ping()`. Three consecutive failures trigger daemon restart. Reset the failure counter on successful ping AND after restart completes. Use `Ping()` — not `refreshTasks()` — for the health check: Ping fails fast, while `refreshTasks()` has two serial RPC calls each with a 5s timeout. Skip the health check when not in daemon mode or already restarting.
- **`AddWriter` must replay before registering.** When adding a writer to a session, send the ring buffer replay BEFORE appending the writer to the `writers` slice. If you register first, `readLoop` can deliver live bytes to the writer before replay is sent, causing duplicate data. The correct order is: snapshot replay → send replay → register writer. This creates a small gap (bytes produced during replay are missed) rather than duplicates. Gaps are handled gracefully by the terminal emulator (partial escape sequences ignored); duplicates cause visible rendering corruption.
- **`ansi.StringWidth` returns 0 for tab characters — expand tabs before width math.** The `charmbracelet/x/ansi.StringWidth("\t")` function returns 0, but terminals render tabs as 1-8 characters (to the next tab stop). Any layout that computes padding, truncation, or column positions from `ansi.StringWidth` will miscalculate if the input contains tabs. The fix: call `expandTabs()` (in `diffparse.go`) to convert tabs to spaces early in the pipeline — before any width calculation, `ansi.Truncate`, or padding math. This applies to diff content (both split and unified views) and potentially any future panel that renders user-provided text.
- **Chroma's terminal formatter resets after every token — use `injectBg` for diff backgrounds.** The `writeToken()` function in chroma emits `\033[0m` (full SGR reset) after each syntax token. This means setting a background color only works for the first token — subsequent tokens lose the background. There is no chroma option to preserve an outer background. The fix is `injectBg(s, bgEsc)` which re-applies the background escape after every `\033[0m` in the string. This is used in both side-by-side diff and unified diff rendering for added/removed line backgrounds.
- **`onFinish` must fire before session removal.** In the runner's exit goroutine, the `onFinish` callback must execute while the session is still in `r.sessions` so that consumers (e.g., daemon exit info cache) are populated before any observer sees the session as gone. The callback runs OUTSIDE `r.mu` to avoid deadlocking if it re-enters the runner (e.g., `HasSession`). The `delete(r.sessions, task.ID)` happens in a second lock section after the callback returns. This ordering prevents a race where `GetExitInfo` returns empty data because the session was deleted before exit info was cached.
- **`startOrAttach` must revert all state on Start failure.** When `runner.Start()` fails, revert `Status` to `Pending`, clear `SessionID`, and zero `StartedAt` before calling `db.Update`. Leaving the task in `InProgress` with no running session creates a ghost task that appears active but has no agent. The `SetStatus(StatusPending)` call doesn't clear `StartedAt` (only `InProgress` and `Complete` have timestamp side effects), so the explicit `t.StartedAt = time.Time{}` is required.
- **All daemon RPC calls must have a timeout.** Go's `net/rpc.Client.Call()` blocks indefinitely with no timeout. If the daemon crashes, every `refreshTasks()` tick (which calls `Running()` + `Idle()` via RPC) hangs the TUI. Use the `c.call()` wrapper (goroutine + select + `time.After`) for all RPC calls. The goroutine may outlive the timeout but will unblock when the OS delivers a connection error on the dead socket. The buffered channel (`make(chan error, 1)`) is critical — without it, the goroutine would block forever on send if the caller already timed out.

- **`IsSandboxAvailable()` is cached via `sync.Once` — safe to call unconditionally.** Checks `os.Stat("/usr/bin/sandbox-exec")` then `exec.LookPath("sandbox-exec")` — effectively free (a single syscall). All subsequent calls return the cached result. The settings view calls it on every refresh to show accurate status. `ResetSandboxCache()` clears the cache (for testing only).
- **sandbox-exec uses SBPL profiles, not JSON config.** `GenerateSandboxConfig()` writes a `.sb` temp file and returns `(profilePath, params []string, cleanup, err)` — a 4-return signature. `params` holds `["HOME=/...", "WORKTREE=/..."]` passed as `-D` flags. `WrapWithSandbox(cmdStr, profilePath, params)` produces `sandbox-exec -D 'HOME=...' -D 'WORKTREE=...' -f profile.sb sh -c 'original cmd'`. The double-sh is intentional (sandbox-exec wraps the shell, which wraps the agent command).
- **SBPL `(allow file-read*)` overrides subsequent `(deny file-read* (subpath X))` ONLY when X is a symlink-resolved path mismatch.** On macOS, `/tmp` is a symlink to `/private/tmp`. A deny rule on `(subpath "/tmp/foo")` does NOT block access to `/private/tmp/foo` because the kernel resolves symlinks before the rule is checked. Always use real, non-symlinked paths in deny rules (e.g., `(string-append (param "HOME") "/.ssh")` where HOME expands to `/Users/username` — a real path). The credential dir protect in the argus profile IS effective because HOME-relative paths are real paths. Never test SBPL rules using `/tmp` paths — use paths under `$HOME` instead.
- **sandbox-exec has no domain-level network filtering.** Unlike srt (which used a proxy to enforce domain allowlists), sandbox-exec's `network*` operations work at the socket/address level. Argus uses `(allow network*)` — all outbound connections are permitted. The sandbox protects FS writes and credential reads, not network egress. This is intentional and documented; do not attempt to add domain filtering via sandbox-exec profiles.
- **The sandbox profile must allow writes to `~/.claude.json` and `~/.claude/`.** Claude Code writes `~/.claude.json` (auth/session state) on every startup. Without this write permission, Claude emits ~41 bytes of terminal init sequences then hangs silently — the agent view appears blank with no output. Also allow `~/.claude/` (subpath) for conversation history writes needed by `--resume`. Blocking either write causes a silent hang, not an error message. The SBPL rules are: `(allow file-write* (literal (string-append (param "HOME") "/.claude.json")))` and `(allow file-write* (subpath (string-append (param "HOME") "/.claude")))`.
- **Symlink-resolved paths bypass subpath write rules.** If `~/.claude/skills` is symlinked to `~/.dots`, the `(allow file-write* (subpath "~/.claude"))` rule does NOT cover writes to `~/.dots/...` — the kernel resolves symlinks before checking. Reads are fine (global `(allow file-read*)` applies to the resolved path). Only add `~/.dots` to the write allow list if agents actually need to write there.
- **The sandbox profile must allow writes to the main repo's `.git` dir for worktree git operations.** Git worktrees store metadata (index, objects, refs) in the main repo's `.git/worktrees/<name>/` directory, not in the worktree itself. Without write access to `<main-repo>/.git`, `git add`, `git commit`, and `git push` all fail with `Operation not permitted`. `resolveGitDir()` in `sandbox.go` reads the worktree's `.git` file (which contains `gitdir: <path>`), walks up two levels from the gitdir path to find the `.git` directory, and adds it to the SBPL write allowlist. This happens automatically in `GenerateSandboxConfig()` — no user configuration needed.
- **Rebuilding the binary does NOT update the running daemon — restart required.** The daemon process loads the binary image at startup and runs it in memory indefinitely. Recompiling `go build -o argus ./cmd/argus/` updates the binary on disk but leaves the daemon process unchanged. Any new tasks started after the rebuild still use the OLD in-memory code (including the OLD `sandboxProfileBase` constant). To pick up sandbox profile changes (or any other daemon-side code changes), the daemon must be restarted: `kill -TERM $(cat ~/.argus/daemon.pid)` — the TUI will auto-restart it via `autoStartDaemon()`. Diagnose staleness by comparing `ls -la /path/to/argus` mtime against `ps -p $(cat ~/.argus/daemon.pid) -o lstart`.
- **Verify the sandbox profile in effect by reading the temp `.sb` file.** The daemon logs the profile path in `~/.argus/daemon.log` as `-f '/var/folders/.../T/argus-sandbox-NNNN.sb'`. While the sandboxed process is still running, `cat <path>` shows the exact SBPL profile being enforced. Compare it against `sandboxProfileBase` in `internal/agent/sandbox.go` to confirm the daemon is using current code. If the profile is missing `~/.claude.json`/`~/.claude/` write rules, the daemon is stale.
- **`sandbox.extra_write` DB entries produce literal SBPL `subpath` rules — garbage input produces broken rules.** The `sandbox.extra_write` config field is CSV-parsed and each value is wrapped in `(allow file-write* (subpath "..."))`. A partial UI entry (e.g., just `"e"`) produces `(allow file-write* (subpath "e"))` — syntactically valid but logically useless. To clear: `sqlite3 ~/.argus/data.sql "UPDATE config SET value='' WHERE key='sandbox.extra_write'"`.
- **Daemon restart must preserve `SessionID` on tasks.** Claude Code's `--session-id` persists conversation state to disk. When the daemon restarts and kills all PTY processes, the restart handler resets tasks to `Pending` but must NOT clear `SessionID`. On re-launch, `startOrAttach` sees `SessionID != ""`, sets `resume = true`, and passes `--resume <id>` to pick up the conversation where it left off. Clearing `SessionID` would lose the conversation history and start a fresh session.

## Development Rules

### Documentation Requirements

- **Every new feature must be documented in `context/knowledge/key-learnings.md` before the session ends.** Add a bullet to the Key Learnings section covering: what the feature does, where the code lives, any non-obvious design decisions, and edge cases handled. If the feature introduces a new pattern (e.g. a new message type, a new DB column, a new key binding), document the pattern so future agents don't have to rediscover it. (Key learnings are imported into CLAUDE.md via `@context/knowledge/key-learnings.md` — do NOT add new learnings directly to CLAUDE.md.)
- **Update `context/knowledge/code-quality.md`** with a dated section summarizing the feature's data model, flow, and any gotchas. Update `context/knowledge/index.md` to include the new key entities.
- **What to document:** new Task fields, new DB columns, new key bindings, new message types, new scan/detection loops, and any edge cases that required a second fix.

### Logging Requirements

- **Every new feature must include uxlog calls for debugging.** All async handlers that process results from external systems (GitHub API, git commands, daemon RPC, etc.) must log both success and failure paths via `uxlog.Log("[feature] ...")`. Use a consistent prefix per feature area (e.g., `[reviews]`, `[git]`, `[daemon]`). Without logs, failures are invisible — the UI just shows blank/stale data with no indication of what went wrong.
- **What to log:** fetch results (count/size), errors, state transitions, and any guards that silently skip work (e.g., cooldown timers, staleness checks).

### Testing Requirements

- **Every change must include tests.** When adding new functionality, fixing bugs, or refactoring, always add or update corresponding tests. Run `go test ./...` to verify all tests pass before considering work complete.
- **Run `go test ./... -cover` after writing tests** to verify coverage improved. Aim for ≥80% coverage on any package you touch.
- **Test file placement:** Tests go in `*_test.go` files in the same package (not `_test` suffix packages). Use the existing `testDB(t)` and similar helpers.
- **What to test:**
  - All exported functions and methods
  - Pure logic functions (parsers, state transitions, builders) — these are easy to test, no excuses
  - View/render functions — verify they produce expected content
  - Edge cases: nil inputs, empty collections, boundary values, error paths
  - State machines: status transitions, cursor navigation, focus cycling
- **What's okay to skip:** Functions requiring a real terminal (raw mode, ioctl), functions that shell out to external processes (git commands, process management), and `cmd/argus/main.go` (entry point)
- **Testing patterns in this codebase:**
  - `db.OpenInMemory()` for database tests (no filesystem needed)
  - `agent.NewRunner(nil)` for UI tests that need a runner but don't start processes
  - `exec.Command("echo", "hello")` or `exec.Command("sleep", "10")` for agent/session tests
  - `DefaultTheme()` for any UI component tests
  - Table-driven tests with `[]struct{ input, expected }` for functions with many cases
  - **Keep daemon client test names short** — macOS Unix socket paths have a 104-byte limit. `t.TempDir()` + long test name exceeds it → `connect: invalid argument`. Use short names like `TestAlive_Dead` not `TestIsSessionAlive_DeadSession`.

## Context Directory

- `context/` stores durable cross-session knowledge checked into git
- `context/knowledge/index.md` is the knowledge graph index — read it when you need project history or domain context
- `context/research/` holds investigation notes and spike results
- `context/plans/` holds strategic plans and proposals

- **`git worktree add` requires a valid local ref or remote-tracking ref as the start point.** If the project config stores just `"master"` but the repo only has `origin/master` (no local branch), `git worktree add -b argus/task <path> master` fails with `fatal: not a valid object name: 'master'`. `resolveStartPoint()` in `worktree.go` resolves this by checking `git rev-parse --verify` and falling back to `upstream/<branch>` then `origin/<branch>`. The new project form also auto-detects the remote default branch (preferring `upstream` over `origin`) when the user enters a repo path, so new projects get the full ref like `upstream/master` by default.

- **Daemon cleanup must run on the `Serve` goroutine, not the `Shutdown` goroutine.** `Shutdown()` is always called from a non-main goroutine (signal handler or RPC handler). After `Shutdown` closes `d.done` and the listener, `Serve()` returns on the main goroutine → `main()` returns → process exits. If `StopAll()` and `removeIfOwnedByPID()` run inside `Shutdown()`, the process may exit before they complete, leaving zombie agent processes and stale socket/PID files. The fix: `Shutdown()` only signals (closes `d.done` + listener); `Serve()`'s accept-loop exit path calls `d.cleanup()` which does `StopAll` + file removal. Additionally, `signal.Stop(sigCh)` must be called after the signal handler goroutine exits — otherwise Go's `signal.Notify` captures SIGTERM into the buffered `sigCh` channel that nobody reads, silently swallowing the signal. This prevents `killExistingDaemon`'s SIGTERM from working, forcing a 2-second wait + SIGKILL escalation every time.

- **The sandbox profile must allow writes to `/var/folders`.** macOS uses `/var/folders` for per-user temp and cache directories (Go's `os.TempDir()`, `NSTemporaryDirectory()`, Node.js temp files, etc.). Without this write permission, agents that write to system temp dirs silently fail or crash. Unlike `/tmp` (a symlink to `/private/tmp`), `/var/folders` is a real path — the rule `(allow file-write* (subpath "/var/folders"))` works as-is with no symlink resolution needed.

- **Agent view file explorer must merge committed + uncommitted changes, not choose one or the other.** The original approach used an if/else: show uncommitted files (`git status --short`) if any, *else* show committed branch files (`git diff --name-status base..HEAD`). This caused blank panels whenever the working tree was momentarily clean between agent commits — the committed changes existed but weren't shown because the uncommitted path returned nothing and the committed path was never reached. Fix: always call `MergeChangedFiles(ParseGitDiffNameStatus(msg.BranchFiles), ParseGitStatus(msg.Status))` which starts with committed branch files as the base and overlays uncommitted status (which takes precedence for the same path). This way the panel always shows all changes on the branch regardless of commit state.

- **File explorer directories auto-expand on cursor movement, one at a time.** `CursorUp()`/`CursorDown()` on `FileExplorer`/`FilePanel` call `autoExpand()` which expands the directory the cursor lands on (or its parent if on a child row) and collapses all others — same pattern as the task list's one-expanded-at-a-time behavior. `autoExpand()` saves `cursorPath` before rebuilding rows and restores the cursor to the same logical row afterward (critical because collapsing a directory shifts row indices). Returns the dir path needing a child fetch (empty string if cached or no dir). Enter on a directory opens the diff view (same as files) — no manual toggle.

- **Per-project sandbox config is stored as three columns on the `projects` table.** `sandbox_enabled` stores `""` (inherit global), `"true"`, or `"false"`. `sandbox_deny_read` and `sandbox_extra_write` store CSV paths appended to the global lists. `ResolveSandboxConfig(task, cfg)` merges global + per-project overrides before calling `GenerateSandboxConfig`. Adding new columns to an existing SQLite table uses `ALTER TABLE projects ADD COLUMN ... DEFAULT ''` — this is called after `CREATE TABLE IF NOT EXISTS` and the error for duplicate column names is silently ignored (covers both new and existing databases). `GenerateSandboxConfig` takes `config.SandboxConfig` directly (not `config.Config`) so the caller controls which merged config is used.

- **PR URL detection scans session output on each tick, not just on finish.** The tick handler calls `runner.Running()` and for each running session scans the last 32KB of output for GitHub PR URLs, avoiding copying the entire ring buffer every tick. Takes the last match so the most recently opened PR wins. A guard prevents redundant DB writes when the URL hasn't changed.

- **Also scan final output on agent finish to catch fast-finishing agents.** `runner.Running()` no longer includes a task once its session exits. If an agent creates a PR and exits before the next 1-second tick, the tick scan misses the URL. Fix: scan the final output on agent finish when no PR URL has been detected yet.

- **The ring buffer must be bounded (256KB), not unbounded.** Setting `defaultBufSize = 0` (unbounded, grow-only) causes CPU spikes and OOM for long-running sessions because `RecentOutput()` copies the entire buffer, and it's called on every tick from multiple hot paths (PR URL scan, agent view render, preview panel). The session log file (`~/.argus/sessions/<taskID>.log`) persists full PTY output to disk for post-restart scrollback, so the in-memory buffer only needs to hold enough for incremental emulator feeding and live scrollback. Use `RecentOutputTail(n)` / `RingBuffer.Tail(n)` when only a small tail is needed (e.g., PR URL scanning only needs last 32KB).

- **Adding a new column to an existing SQLite table uses `ALTER TABLE ... ADD COLUMN ... DEFAULT ''` after `CREATE TABLE IF NOT EXISTS`.** The error for a duplicate column (column already exists) is silently ignored with `//nolint:errcheck`. This pattern covers both fresh databases (CREATE TABLE includes the column) and existing ones (ALTER TABLE adds it). Used for `pr_url` on tasks and `sandbox_enabled/deny_read/extra_write` on projects. Any future column additions must follow this same pattern in `createTables()` — never use schema migrations for simple column additions.

- **`taskColumns` is the canonical column list used in every task query.** When adding a new column to the tasks table, update `taskColumns`, `scanTask` (add `&t.NewField` in the exact same position), `Add` (insert/values), and `Update` (SET clause) all in lockstep. Missing any one of these causes silent data loss or scan errors that only manifest at runtime.

- **Use `prURLRe.FindAllString(output, -1)` and take `matches[len(matches)-1]` for PR URL detection.** Taking the last match means the most recently opened PR wins — correct when an agent opens a draft, revises it, and opens a final PR. The regex `https://github\.com/[a-zA-Z0-9_.\-]+/[a-zA-Z0-9_.\-]+/pull/\d+` covers both plain-text URLs and OSC 8 hyperlinks (where the URL appears twice — as target and display text). The final match is the display text occurrence.

- **`'o'` key to open PR in browser from task list uses `exec.Command("open", url).Start()`.** The `open` command on macOS opens any URL in the default browser. The command must run asynchronously so it doesn't block the UI.

- **`gh search prs --json` does NOT support `reviewDecision` — use `gh pr list` instead.** The GitHub Search API returns `SearchResultItem`, not `PullRequest`. `reviewDecision` is available via `gh pr list --json` and `gh pr view --json` (both use the GraphQL PullRequest type). `FetchPRList()` in `internal/github/github.go` fetches the cross-repo list via `gh search prs`, then enriches with `reviewDecision` via `gh pr list` grouped by repo — O(repos) not O(PRs).

- **Reviews tab: `internal/tui2/reviews.go` + `internal/github/github.go`.** Three-panel layout (PR list / diff / comments). PR list fetched via `gh search prs`, files via REST API, full diff via `gh pr diff` (cached per PR, re-sliced per file via `ExtractFileDiff`). Comments via REST. Review actions: `a` approve, `r` request changes, `c` line comment. Focus cycles: list → diff → comment compose. Auto-refresh on tick: comments have 2min TTL, diff staleness checked against `PR.UpdatedAt`. All message handlers log to uxlog with `[reviews]` prefix. **`SetPRs` must sort review requests before "my PRs" in the slice** — the PR list renders "Review Requests" first and "My Open PRs" second, but the cursor navigates the flat slice sequentially. If the slice order doesn't match the visual order, the cursor starts in the wrong section and review requests appear unreachable. **PR list caching:** `prListCooldown` (10min) gates refetch on tab entry. `SetPRs` preserves cursor, scroll offset, and selection on background refreshes (only resets on first load). Shows cached data during background refresh with a dimmed "refreshing…" indicator; errors show as "refresh failed: ..." rather than replacing the list. `'R'` forces a manual refresh subject to the same cooldown.

- **Task rename is display-only — worktree dir and branch are unchanged.** The 'r' key in the task list opens a rename form. Rename only updates `t.Name` in the DB — the worktree directory path, git branch, and session ID are all preserved. This means rename works even while an agent is running.

- **Codex resume is encoded in argus, not user-configurable.** `Backend.ResumeCommand` was removed. - **KB (Knowledge Base) is a SQLite FTS5 full-text search store baked into the Argus daemon.** `internal/kb/` defines `Document`, `SearchResult`, and `KBStore` interface. `internal/db/kb.go` implements `KBStore` on `*db.DB` using FTS5 (`kb_documents` virtual table) and a companion `kb_metadata` table for non-text fields (timestamps, word count). - **FTS5 virtual table creation uses `CREATE VIRTUAL TABLE IF NOT EXISTS ... USING fts5(...)` — not a regular CREATE TABLE.** The `USING fts5(...)` clause selects the FTS5 module. Columns marked `UNINDEXED` (path, tier) are stored but not indexed for full-text search. The `snippet()` function in FTS5 queries takes `(table, column_index, highlight_start, highlight_end, ellipsis, max_tokens)` — column indices are 0-based and count only the non-UNINDEXED columns differently depending on the FTS5 version. Use `ORDER BY rank` for BM25 ranking (lower rank = better match). The FTS5 `MATCH` operator requires sanitized queries — strip `" * ( ) : ^ { }` characters using `kb.SanitizeQuery()`.

- **Obsidian task syntax parsing (`internal/import/obsidian.go`) requires `#argus` to be mandatory in the regex.** Using `(?:\s+#argus(...)?)?$` (optional `#argus`) causes the regex to match ALL unchecked task lines — not just those tagged `#argus`. The correct pattern makes `\s+#argus` required: `^- \[ \] (.+?)\s+#argus(?:/([^\s]+))?(?:\s.*)?$`. The `(?:\s.*)?$` suffix allows trailing text after the project tag. - **FTS5 `SanitizeQuery` must strip all operator characters including `-` and `+`.** `-` is the FTS5 NOT operator and `+` is the proximity operator. Full set to strip: `" * ( ) : ^ { } - +`. Omitting `-` or `+` allows users (or agents via `kb_ingest`) to inject FTS5 query syntax, causing unexpected search results or errors. The canonical list is in `internal/kb/search.go`.

- **FTS5 + metadata JOIN avoids N+1 under mutex.** `KBSearch` must not issue a per-row `d.conn.QueryRow` inside the `rows.Next()` loop while `d.mu` is held. The nested `QueryRow` also needs `d.mu` → deadlock. Fix: use `LEFT JOIN kb_metadata km ON km.path = kb_documents.path` with `COALESCE(km.modified_at, 0)` in the main query. Same pattern as `KBList`. - **Never call `GetInnerRect()` or any tview widget method from the tick goroutine.** tview is not thread-safe — widget state is set during `Draw()` on the main goroutine. Cross-goroutine reads cause data races. The fix for `SyncPTYSize`: `Draw()` computes the desired PTY size under `tp.mu` and stores it in `pendingResizeRows`/`pendingResizeCols`. The tick goroutine atomically reads and clears the pending values under the same lock, then performs the resize RPC. This keeps all tview access on the main goroutine and all RPC calls off it.

- **`refreshTasks()` must not do RPC while holding `a.mu`.** `a.runner.Running()` is an RPC call (up to 5s timeout). Calling it inside `a.mu.Lock()` blocks the entire UI — any `QueueUpdateDraw` callback that also needs `a.mu` is stalled, causing a visible freeze. Fix: fetch `runningIDs` OUTSIDE the lock (same pattern as `onTick`), then lock and call `refreshTasksWithIDs(runningIDs)`.

- **Daemon session exits must be wired to the tui2 app via `client.OnSessionExit`.** Without this callback, when an agent process exits in daemon mode, the TUI never learns about it — `task.Status` stays `InProgress` forever and the agent view shows stale state. The callback fires `HandleSessionExit(taskID, info)` which updates the task to `Complete` (or `Pending` if stopped) via `QueueUpdateDraw`. For `StreamLost` exits, status is preserved since the process may still be alive.

- **x/vt `SafeEmulator` hangs on terminal query sequences — use `newDrainedEmulator()`.** The `charmbracelet/x/vt` emulator uses `io.Pipe()` internally for terminal responses. When processing DA1 (`\x1b[c`), DA2 (`\x1b[>c`), DSR (`\x1b[6n`, `\x1b[5n`), and similar query sequences, the emulator writes responses to the pipe writer which blocks indefinitely because nobody reads the pipe reader. This causes `emu.Write()` to hang, blocking the tview main goroutine and freezing the entire UI. Claude Code and Codex both emit these sequences to detect terminal capabilities. The fix: `newDrainedEmulator(cols, rows)` in `terminalpane.go` creates the emulator and immediately starts `go io.Copy(io.Discard, emu)` to drain the response pipe. This handles ALL query sequences generically — no need to strip specific escape sequences from the input. Both `TerminalPane` (live + replay rendering) and `TaskPreviewPanel` (task list preview) use `newDrainedEmulator()`. The drain goroutine exits when the emulator is garbage collected. Never use bare `xvt.NewSafeEmulator()` in the tui2 package.

But pressing Ctrl+D again in the agent view did nothing — `tcellKeyToBytes` converts it to `{0x04}` which reaches the "Forward to PTY" block, but with no alive session the key is silently dropped. Fix: in `handleAgentKey`, check for `KeyCtrlD` when `sess == nil || !sess.Alive()` and call `exitAgentView()`. This mirrors the existing `'o'` key guard pattern for dead sessions.

- **`TerminalPane.MouseHandler()` enables mouse wheel scrolling in the agent view.** tview dispatches mouse events via `MouseHandler()` on primitives — the default `Box.MouseHandler()` does nothing with scroll events. `TerminalPane` overrides it using `WrapMouseHandler` (which checks bounds) to handle `tview.MouseScrollUp` and `tview.MouseScrollDown`. Additionally, `SyncPTYSize()` runs in the 200ms `startAgentRedrawLoop` (not just the 1-second tick) to minimize the window where the PTY size is stale.

- **x/vt emulator can panic on replay data from differently-sized terminals — wrap writes with `safeEmuWrite`.** When PTY output recorded at e.g. 82 rows is replayed into a smaller emulator (e.g. 59 rows), cursor positioning sequences (`CSI 82;1H`) followed by scroll operations (`ESC M` reverse index) cause `InsertLineArea` to panic with index-out-of-range. This is an upstream bug in `charmbracelet/x/vt` — the emulator doesn't clamp cursor positions to buffer bounds before scroll operations. `safeEmuWrite()` in `terminalpane.go` wraps `emu.Write()` with `recover()` and logs the panic via uxlog. Fix: `refreshPreview` in `app.go` calls `sess.PTYSize()` for live sessions and passes the PTY's column count as the emulator width. `Draw()` already clips `renderCols = min(cellCols, inner.W)`, so the wider cell grid is painted correctly within the narrow panel. - **Agent view replay must use current panel dimensions, not stale PTY size.** When a session dies, `TerminalPane.ptyCols/ptyRows` retains the last live PTY size. If the user resizes the window after the session finishes, replay renders at the old width — clipping or leaving gaps. Fix: in `Draw()`, for dead/nil sessions, override `ptyCols/ptyRows` with `wantCols/wantRows` (current panel dimensions) so `renderReplay` creates the emulator at the current size. The replay path already creates a fresh emulator on every `Draw()` call, so this adds no overhead.

- **Agent view has a powerline-style header showing the current task name.** `AgentHeader` (`internal/tui2/agentheader.go`) is a 1-row `tview.Box` widget that renders a centered powerline segment with the task name using the same color palette as the root `Header` (lavender active segment on dark background). - **Live scrollback reads from the session log file, not just the ring buffer.** The 256KB in-memory ring buffer limits scrollback for live sessions. When `scrollOffset > 0`, `Draw()` calls `readLogTail(needed)` which seeks from EOF in `~/.argus/sessions/<taskID>.log` — the same file that `readLoop` appends all PTY output to unboundedly. The read size is estimated as `(scrollOffset + viewportHeight) * cols * 3` (3x for ANSI overhead), with a 1MB minimum. This gives effectively infinite scrollback while keeping the live follow-tail path (scrollOffset == 0) on the fast ring buffer path. `readLogTail` uses `os.Open` + `ReadAt` (not `ReadFile`) to safely read a file being concurrently appended. vt10x handles truncated escape sequences at the read boundary gracefully. Falls back to ring buffer if the log file doesn't exist.

- **Anchor-lock keeps scrolled-up content pinned when new output arrives (tmux-style).** `paintEmu` tracks `anchorTotalLines` — the total addressable lines (scrollback + visible) when the scroll position was last rendered. When `totalLines` grows (new output arrived), `scrollOffset` is bumped by the delta so the viewed content stays at the same absolute position. Without this, new lines push the viewport content upward while the user is reading. `anchorTotalLines` is reset to 0 when scrolling back to bottom (`ScrollDown` past 0 or `ResetScroll`) so it doesn't interfere with follow-tail mode.

- **Replay emulator is cached and reused when only scroll offset changes.** `renderReplay` previously created a fresh `xvt.SafeEmulator` and replayed the entire buffer on every frame. Now it caches the emulator in `replayEmu` and only rebuilds when the input changes (different `logSize` for log-backed reads, different `len(raw)` for ring buffer fallback, or different PTY dimensions). When only `scrollOffset` changes, `paintEmu` reuses the cached emulator and just re-paints with the new viewport window. This avoids O(buffer_size) replay on every scroll step. Cache is cleared on `ResetVT()` and task switch.

- **`ensureCursorVisible` must reset scrollOffset when all lines fit.** The original implementation took `innerW` as the first parameter and computed the cursor line independently. When text wrapped from 1 to 2 lines, the intermediate state (visiblePromptLines=1, cursorLine=1) set scrollOffset=1. The next frame with visiblePromptLines=2 didn't reset it — line 0 disappeared. Fix: the first parameter is now `totalLines` instead of `innerW`. When `totalLines <= visibleLines`, scrollOffset is unconditionally reset to 0. This mirrors the old Bubble Tea fix which set height to `maxPromptLines` before `Update()` to prevent `repositionView()` from scrolling.

- **tui2 `NewTaskForm` has word navigation via `wordboundary.go` (ported from old `internal/ui/wordboundary.go`).** Alt+Left/Alt+B jumps word left, Alt+Right/Alt+F jumps word right, Alt+Backspace/Ctrl+W deletes word left, Alt+Delete/Alt+D deletes word right. Word boundaries use zsh WORDCHARS semantics (`*?_[]~=&;!#$%^(){}<>` are word chars; `/`, `-`, `.` are delimiters). tcell reports Alt+arrow as `tcell.KeyLeft`/`KeyRight` with `tcell.ModAlt` modifier; Alt+rune as `tcell.KeyRune` with `tcell.ModAlt` and the rune value.

- **tui2 `NewTaskForm` has `/skill` autocomplete (ported from old `internal/ui/newtask.go`).** State: `acOpen`, `acMatches`, `acIdx`, `acScroll`. Skills loaded via `skills.LoadSkills()` on form creation and on project change. Trigger is `/` for Claude backends, `$` for Codex (via `agent.IsCodexBackend()`). Autocomplete opens when prompt starts with trigger and has no spaces. Up/Down navigate with wrapping; Enter selects (injects `/<name> `); Esc closes dropdown without canceling form; space closes (user typing args). `acMaxVisible=6` items shown with scroll indicator.

- **Project headers in the task list show an aggregated status icon with chevron and task count.** `drawProjectRow` in `internal/tui2/tasklist.go` renders: icon + chevron + project name + (count). `projectStatusIcon()` computes the icon from all tasks in that project with priority: in_progress > in_review > all_complete > mixed > all_pending. When all in-progress tasks are idle, shows a moon icon (☾). The `tickEven` field toggles on each tick for icon animation (alternates between ● and ◉ for active in-progress). `refreshTasksWithIDs` in `app.go` passes both `runningIDs` and `idleIDs` from the runner, calling `SetIdle()` and `Tick()` on every refresh cycle.

- **Daemon restart from Settings tab via `srDaemon` row.** When `daemonConnected` is true, the Status section shows a "Restart Daemon" row (`srDaemon` kind). Enter triggers `OnRestartDaemon` callback → `App.restartDaemon()` in a goroutine. During restart, the row label changes to "Restarting..." and Enter is a no-op. `SetDaemonRestarting(false)` is called from `restartDaemon()`'s success and error `QueueUpdateDraw` callbacks to reset the label. The detail panel shows daemon status and `[enter] restart daemon` hint. The row only appears in daemon mode — not in in-process mode.

- **Stale in-progress tasks are reconciled to complete on every tick (daemon mode only).** `refreshTasksWithIDs` checks all tasks: if `StatusInProgress` but not in the daemon's running set, mark `Complete`. This catches cases where the `handleSessionExitUI` callback didn't fire (daemon restart with stale binary, TUI restart mid-session, stream loss without proper exit). Only runs when `daemonConnected` is true — in-process mode has its own `onFinish` callback. `restartDaemon` also wires `OnSessionExit` on the new client (was previously missing, causing all sessions started after a daemon restart to never fire exit callbacks).

- **Stopped agent → `StatusInReview`, not Pending.** `handleSessionExitUI` in `app.go` sets `StatusInReview` when `stopped == true` (user explicitly stopped the agent). Normal clean exit → `StatusComplete`. This matches the pre-tcell Bubble Tea behavior: stopped means "needs human review", not "reset to pending".

- **Idle+unvisited tasks are visually promoted to InReview in the task list.** `App` tracks `idleUnvisited` and `viewedWhileAgent` maps (same as pre-tcell). When an in-progress task becomes idle and the user hasn't visited its agent view since, `drawTaskRow` renders it with the InReview icon (◎) and cyan style. `projectStatusIcon` also counts these as InReview at the project level. Entering the agent view (`onTaskSelect`) clears the flag. `refreshTasksWithIDs` computes newly-idle tasks by diffing against `IdleSet()` (the previous idle map) and adds them to `idleUnvisited`; tasks no longer idle are removed. `viewedWhileAgent` suppresses false-positive re-adds for tasks the user was watching when they went idle.

- **`s`/`S` keys cycle task status manually.** `s` advances via `Status.Next()`, `S` reverts via `Status.Prev()`. Wired through `OnStatusChange` callback → `db.Update` + `refreshTasks`. This allows manually cycling through all four states: Pending → InProgress → InReview → Complete.

- **Task row icon animates between ● and ◉ for actively running tasks.** `drawTaskRow` checks `tickEven` for InProgress tasks that are running and not idle, alternating the icon on each tick. Idle tasks (visited) show moon (☾). Idle+unvisited show InReview icon (◎). This matches the pre-tcell 4-way branch: idleUnvisited → running/idle → running+tickEven → running+!tickEven.
