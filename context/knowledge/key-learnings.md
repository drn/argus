## Key Learnings

- **`ctrl+q` in diff mode must both exit diff mode AND refocus the agent panel.** When a file diff is open (`diffMode == true`), pressing `ctrl+q` calls `exitDiffMode()` but must also set `av.focus = panelAgent` — matching the behavior of the `esc` key in `handleDiffKey`. Without `av.focus = panelAgent`, the user would land back on `panelFiles` and need a second keypress to reach the terminal. The global `ctrl+q` handler in `HandleKey` (agentview.go) runs before `handleDiffKey`, so the fix belongs there: `exitDiffMode(); av.focus = panelAgent; return false, nil`.

- **`findInputRow` decouples the input highlight row from the cursor parking row.** Claude (Ink-based) parks the hardware cursor at an empty trailing row after each render — `cur.Y` is not the input row. `renderLine` now takes `isInputRow bool` separately from `cursorX`, so the blue `activeInputBG` highlight can land on the correct row. `findInputRow(vt, curY, cols)` scans upward from `curY`: it first looks for a row containing `❯` (Claude Code's prompt marker, `U+276F`) which precisely identifies the input row even when a tip/hint row sits between input and cursor. If no `❯` is found, it falls back to the last non-empty row (correct for Codex, which has no prompt marker but places the cursor directly on the input). Both `renderIncremental` (agentview.go) and `replayVT10X` (vtrender.go) compute `inputRow = findInputRow(...)` before the render loop and pass `y == inputRow` as `isInputRow` to `renderLine`.

- **The trailing-empty-line trimmer must protect the cursor line.** `stripANSI` removes all ANSI escape codes and `TrimSpace` collapses what remains. A cursor cell is just a space with a colored background — after stripping it becomes `""`, so the trim loop silently removes the cursor line and hides the cursor. This is the primary reason the cursor appears "invisible" or "black" in the agent view. The fix in both `renderIncremental` (agentview.go) and `replayVT10X` (vtrender.go): add a guard `len(lines)-1 != cur.Y` to the trailing trim loop so the cursor line is never trimmed. Apply the same guard to the leading trim using a `frontRemoved` counter. TUI agents like Claude Code (Ink-based) park the cursor at an empty bottom row after each render — without this guard, every render cycle removes that row and the cursor never appears.

- **Opening a PR from agent view: `ctrl+p` when session active, `o` when finished.** Use `ctrl+p` (`tea.KeyCtrlP`, raw byte 0x10) to open the PR URL in the browser — works in every terminal without configuration. Avoid modifier+key shortcuts (Cmd+O, Option+O) — macOS terminals intercept ⌘+key at the app level, and Option+O sends `ø` (U+00F8) unless the terminal is configured with "Use Option as Meta key" which most users don't have enabled. `ctrl+p` is intercepted in `handleAgentViewKey` in `root.go` BEFORE the key reaches the PTY, so it works while the agent is running. Plain `o` is also intercepted when `m.runner.Get(m.agentview.taskID) == nil` (session finished, PTY is dead). Status bar hint shows `^p` when active, `o` when inactive. The PR URL is stored in `AgentView.taskPRURL` (set via `SetPRURL`). Three places keep it current: (1) `enterAgentView` looks up the task from DB at view entry; (2) `PRDetectedMsg` handler pushes new URLs while the agent view is active; (3) `handleAgentFinished` pushes URLs discovered from the final output.

- **New task form has `/skill` (or `$skill` for codex) autocomplete with project-specific skills.** `internal/ui/skills.go` scans `~/.claude/skills/` AND the selected project's `<projectPath>/.claude/skills/` for skill directories. Project-specific skills take precedence on name collision (extraDirs are searched first). The trigger character is `/` for claude backends and `$` for codex backends — determined by `acTrigger()` which calls `agent.IsCodexBackend()` on the selected backend's command. When the prompt value starts with the trigger and contains no spaces, a filtered dropdown appears. Changing the project selection fires `loadSkillsCmd(projectPath)` to reload skills asynchronously. Changing the backend calls `updateAutocomplete()` to recheck the trigger and close/reopen the dropdown appropriately. `root.go` also passes the initial project path to `loadSkillsCmd` when opening the form.

- **Async skill loading must refresh autocomplete state on delivery.** The new task form loads skills via `loadSkillsCmd()` in a background goroutine. If the user types `/de` before the scan finishes, `updateAutocomplete()` runs against an empty skill list and closes the dropdown. When `skillsLoadedMsg` arrives, the root model must both assign `m.newtask.skills` and immediately call `m.newtask.updateAutocomplete()`; otherwise the dropdown stays closed until the next keystroke, making autocomplete feel flaky on slower filesystems.

- **New task backend selector pre-selects the global default backend — no inherit option.** The selector shows only the sorted backend names (no `(inherit)` entry). `NewNewTaskForm` finds the index of `defaultBackend` in the sorted list and sets `backendIdx` to it, falling back to index 0. `SelectedBackend()` always returns an explicit name; `Task().Backend` is never empty. Backend resolution (`task.Backend > project.Backend > global.Backend`) is not relevant here — the form always stores an explicit choice.

- PTY child processes need a real terminal size at launch (`pty.StartWithSize`), not 0x0. TUI apps like claude won't render with zero dimensions.
- Use `charmbracelet/x/term` for raw mode (cross-platform) instead of platform-specific ioctls (`TIOCGETA` is macOS-only, `TCGETS` is Linux-only).
- `tea.ExecCommand.SetStdin/SetStdout` must capture and use Bubble Tea's `p.input`/`p.output` — don't hardcode `os.Stdin`/`os.Stdout`.
- The single-reader-tee pattern (one goroutine reads PTY, tees to buffer + optional writer) is critical. Two goroutines reading the same fd causes data loss.
- **Never run git commands or filesystem discovery synchronously in `Update()`.** Even fast git commands take 50-500ms which freezes the entire TUI. Use `tea.Cmd` to run them in background goroutines and deliver results via messages. Cache resolved paths in a `map` on the model to avoid repeated lookups.
- **Backend default config must be self-healing.** The default claude backend command MUST include `--dangerously-skip-permissions`, MUST NOT use `-p` as the prompt flag, and MUST NOT include `--worktree` (Argus manages worktrees itself via `git worktree add`). `-p` puts Claude in non-interactive print mode (process prompt → exit), which silently breaks agent sessions. The `fixupBackends()` method in `internal/db/migrate.go` runs on every `Open()` to detect and repair outdated backend configs. Any change to `DefaultConfig()` backend values must be mirrored in `fixupBackends()` detection logic, or existing databases will retain stale values.
- **Use incremental vt10x feeding, not full replay.** The agent view's terminal panel uses a persistent `vt10x.Terminal` that receives only new bytes each tick (`renderIncremental`), not the entire 256KB ring buffer. Full replay is O(buffer_size) and causes input lag when the buffer is large. The persistent terminal is reset on resize, task switch, or ring buffer wrap. The full-replay path (`formatTerminalOutput`) is only used for scrollback mode and finished sessions.
- **Sub-view structs that need mutation must be pointers.** Bubble Tea's `Update` uses a value receiver, so helper methods with value receivers (like `scheduleGitRefresh`) get a copy of the model. Mutations to value-type fields inside those helpers are silently lost. Any sub-view struct that is mutated outside the direct `Update` method body must be stored as a pointer (e.g., `gitstatus *GitStatus`, `agentview *AgentView`). Fields that are only read (or only mutated directly in `Update`) can stay as values. Maps (`resolvedDirs`) are already reference types and work correctly.
- **`keyMsgToBytes` must preserve the Alt modifier for all key types.** When converting `tea.KeyMsg` to raw terminal bytes, check `msg.Alt` and prepend ESC (`0x1b`) for every key category — runes, arrows, and special keys (`keyByteMap`). Dropping Alt silently breaks macOS Option+Delete (word delete), Option+arrows (word movement), and other Alt-modified shortcuts. The `altArrowMap` handles arrows with dedicated CSI sequences; all other keys use ESC-prefixed encoding.
- **Use `msg.Type` not `msg.String()` for Bubble Tea key matching.** String comparison fails when terminals encode modifier keys differently (e.g., urxvt sends `\x1b[Od` for ctrl+left, which Bubble Tea parses as `KeyCtrlLeft` with `Alt=true` → `String()` returns `"alt+ctrl+left"` instead of `"ctrl+left"`). Always match on `msg.Type` and check `msg.Alt` separately. For agent view pane switching, Cmd+left/right (sent as Alt+left/right by macOS terminals) is the binding — the git status panel is not focusable, so focus toggles between terminal and files only. Plain left/right arrows are passed through to the agent process.
- **`textarea.Model` panics when zero-valued; `textinput.Model` does not.** The `textarea.Model` has internal pointers (`viewport`, `style`) that are nil at zero value — calling `SetWidth` or `SetHeight` on an uninitialized textarea causes a nil pointer dereference. Any form struct using `textarea.Model` must guard `SetSize`-like methods against being called before the constructor runs (e.g., check a non-nil field like `projects`). This matters because root model's `WindowSizeMsg` handler calls `SetSize` on all sub-views, including zero-valued ones that haven't been opened yet.
- **`removeWorktree` must validate paths before `os.RemoveAll`.** The `os.RemoveAll` fallback in `removeWorktree` will nuke any directory if `git worktree remove` fails — including the root project if `t.Worktree` is incorrectly set. The `isWorktreeSubdir` guard ensures the path contains `/.argus/worktrees/` or `/.claude/worktrees/` before any removal. Any new cleanup code that calls `os.RemoveAll` on user-provided paths must have a similar safety check.
- **Use `msg.Button` not `msg.Type` for Bubble Tea mouse events.** The `tea.MouseMsg.Type` field and constants like `tea.MouseWheelUp` are deprecated in bubbletea v1.3+. Use `msg.Button` with `tea.MouseButtonWheelUp`/`tea.MouseButtonWheelDown` instead. Mouse events are routed through root `Update()` as `tea.MouseMsg` — the root dispatches to the appropriate sub-view based on `m.current`.
- **Don't clamp scroll offset against empty cached state.** The agent view's `scrollUp` must not clamp `scrollOffset` to 0 when `cachedLines` is empty. In incremental render mode, `cachedLines` is only populated by `formatTerminalOutput`, which is only called when `scrollOffset > 0`. Clamping to 0 when the cache is empty creates a chicken-and-egg deadlock where scrolling can never start. Let the offset grow freely when the cache is empty — the render path handles over-scroll gracefully via `windowLines`, and subsequent scrolls will clamp correctly once `cachedLines` is populated.
- **`textarea.LineCount()` only counts hard newlines, not soft wraps.** A long line that visually wraps to 3 lines still reports `LineCount() == 1`. For auto-resize modals with single hard lines (enter disabled), use `LineInfo().Height` which returns the exact wrapped line count computed by the textarea's own internal `memoizedWrap` — no reimplementation needed. Do NOT use character-count division (`runeLen/width`); the textarea uses word wrapping (breaks at word boundaries), so character-based counting underestimates. For multi-line pasted content, fall back to `maxPromptLines`. Also use `LineInfo().RowOffset` and `.Height` to detect when the cursor is at the first/last visual line for arrow-key navigation out of the textarea.
- **Use `ansi.Hardwrap` not `ansi.Truncate` for panel content that should wrap.** The `charmbracelet/x/ansi` package provides both: `Truncate` clips lines at a width (good for single-line labels), while `Hardwrap` inserts newlines at the width boundary preserving ANSI escape sequences (good for multi-line content panels like diff views). When using `Hardwrap`, pre-compute and cache the wrapped lines (`diffWrappedLines`/`diffWrapWidth` pattern) since wrapping is O(content_size) and the width only changes on resize. Invalidate the cache when content changes or width changes. Scrolling must operate on visual (wrapped) lines, not source lines.
- **`textarea.SetHeight()` does not reset the viewport scroll offset.** The textarea's `Update()` calls `repositionView()` at the end, which scrolls the viewport to keep the cursor visible based on the *current* viewport height. If the viewport height is too small (e.g., 1), it scrolls down to follow the cursor. `SetHeight()` only changes `viewport.Height` — it does NOT call `repositionView()`, so the stale `YOffset` persists. For auto-resizing textareas, always expand the height to the maximum BEFORE calling `textarea.Update()`, then shrink it back to the actual needed height afterward. This prevents `repositionView()` from scrolling unnecessarily. Also note: the textarea's `wrap()` function uses `>=` (not `>`) for width comparison, so text that exactly fills the width will wrap to the next line.
- **`⌘` (U+2318) is double-width in terminals but `go-runewidth` reports 1.** Any status bar or layout that includes `⌘` must add `strings.Count(s, "⌘")` to the `lipgloss.Width()` result to compensate. Without this, gap math underestimates the rendered width and the bar overflows. Check for similar mismatches when adding other Unicode symbols to the UI — test with `runewidth.RuneWidth()` and compare against actual terminal rendering.
- **All `View()` paths must handle zero dimensions without panicking.** Bubble Tea calls `View()` before delivering the first `WindowSizeMsg`, so `m.width` and `m.height` are both 0 on the initial render. Any arithmetic like `m.height - N` produces a negative value — passing that to slice expressions or layout functions causes panics. Every function that receives a computed height/width must guard `<= 0` at the top (return empty string or no-op). The `padToBottom` helper already had this guard; `padHeight` and any future layout helpers must too.
- **Do not use either bare reverse-video or hard-coded black for the vt10x cursor inside Lipgloss panels.** Bare `\x1b[0;7m` depends on the host terminal defaults, while hard-coding black (`40`) makes the cursor disappear against Argus's dark panel background. The robust fix in `renderLine` is: (1) give the cursor cell its own explicit high-contrast palette, independent of the source cell attributes, and (2) if you want the active input row to read like an input field, only tint cells that still use `vt10x.DefaultBG`. That preserves Codex/Claude colors and explicit cell backgrounds while still separating the editable row from surrounding output.
- **Always render the cursor regardless of `CursorVisible()`.** TUI agents like Claude Code (built with Ink) hide the hardware cursor via `\x1b[?25l` — standard practice for TUI apps that manage their own rendering. The vt10x library correctly tracks this, so `CursorVisible()` returns `false`. If `renderLine`'s `cursorX` is gated on `CursorVisible()`, the `\x1b[0;7m` cursor block is never emitted and the cursor appears absent (black/invisible). Both `renderIncremental` and `replayVT10X` must pass the cursor position unconditionally — `CursorVisible()` is irrelevant for Argus's use case since the agent view always wants to show the cursor position.
- **Self-perpetuating tick chains must have exactly one starting point.** A `tea.Tick` handler that schedules the next tick creates a self-sustaining chain. If *any other* handler also schedules that same tick type (e.g., a slower parent tick, a view-enter function), you get N parallel chains accumulating over time — each one triggers a full `View()` re-render, causing progressive lag that grows linearly with time spent in the view. **Rule:** each tick message type should be started from exactly one place (typically view entry), and the self-perpetuating handler is the only thing that reschedules it. Audit all `tea.Tick` calls in the codebase when adding a new tick type to ensure no accidental duplication. The symptom of this bug is progressive input lag that temporarily clears when switching views (accumulated tick messages drain from the queue).

- **All panels in `PanelLayout.Render()` must enforce their own width.** `PanelLayout.Render()` only pads height and joins horizontally — it does NOT enforce column widths. If a panel's content is narrower than its allocated width, the panel collapses to content width and the layout breaks visually. Every panel passed to `Render()` must use `borderedPanel(width, height, ...)` or `lipgloss.NewStyle().Width(w)` to fill its allocation. The agent view panels already do this via `borderedPanel`; the task list view's left pane was missing it.
- **Worktree creation must succeed before task persistence.** Never `db.Add(task)` without a valid worktree. If `CreateWorktree` fails, keep the new task form open with the error — do NOT fall back to running in the project directory or delegating to `--worktree`. The `ResolveTaskDirMsg` handler must only persist paths that pass `isWorktreeSubdir()` to prevent project directories from being saved as `t.Worktree`. `CreateWorktree` returns `(wtPath, finalName, err)` — the `finalName` may differ from the input if name conflicts required a `-1`, `-2` suffix.
- **Map lookups returning `*T` become non-nil interfaces.** When a method returns an interface (e.g., `Get(id) SessionHandle`) and the underlying implementation uses a map (`sessions[id]`), a missing key returns `nil *Session`. Assigning this to an interface gives a **non-nil interface** with a nil concrete value — `== nil` checks fail. The `Get` method must explicitly check `if sess == nil { return nil }` before returning. This applies to any method on Runner/Client that returns `SessionHandle`.
- **Stream loss ≠ process exit — use `StreamLost` flag to distinguish.** When a stream connection fails, `connectStream` routes to one of three paths: (1) **process exited**: call `removeSession` (queries `GetExitInfo`, fires `AgentFinishedMsg` → normal exit flow); (2) **stream lost / process alive**: call `removeSessionStreamLost` → `AgentFinishedMsg{StreamLost: true}` → TUI keeps task `InProgress` and shows a warning; (3) **daemon unreachable**: also call `removeSessionStreamLost` — can't confirm process exit, so keep `InProgress`. `isSessionAlive()` returns `(alive bool, reachable bool)` — when `reachable=false` (RPC failed), treat as stream lost, not process exit. Retry exhaustion (3 retries × 500ms) is also stream lost, not process exit. `handleAgentFinished` returns early when `msg.StreamLost=true`, skipping `determinePostExitStatus` entirely.
- **`client.Get()` must return `nil` for any non-alive session.** The original condition `!info.Alive && info.PID == 0` is too lenient. During the window between `onFinish()` firing and `delete(r.sessions, taskID)` executing in the runner, `SessionStatus` returns `{Alive: false, PID: non-zero}` (process has exited but the session hasn't been removed yet). The lenient check lets `Get()` create a new `RemoteSession` with its own `connectStream` goroutine, firing a second `AgentFinishedMsg` for the same task. Fix: use `!info.Alive` alone — the PID is a stale artifact. A dead session must never be returned as a valid handle regardless of PID.
- **Daemon health check via `daemonFailures` counter in TickMsg.** The TickMsg handler pings the daemon on each tick using `client.Ping()` (type-assert `m.runner` to `*dclient.Client`). Three consecutive failures set `m.daemonRestarting = true` and trigger `restartDaemonCmd()`. Reset `m.daemonFailures = 0` on successful ping AND in the `DaemonRestartedMsg` handler. Use `Ping()` — not `refreshTasks()` — for the health check: Ping fails fast, while `refreshTasks()` has two serial RPC calls each with a 5s timeout. Skip the health check when `!m.daemonConnected` (in-process mode) or `m.daemonRestarting` (already restarting).
- **`AddWriter` must replay before registering.** When adding a writer to a session, send the ring buffer replay BEFORE appending the writer to the `writers` slice. If you register first, `readLoop` can deliver live bytes to the writer before replay is sent, causing duplicate data. The correct order is: snapshot replay → send replay → register writer. This creates a small gap (bytes produced during replay are missed) rather than duplicates. Gaps are handled gracefully by vt10x (partial escape sequences ignored); duplicates cause visible rendering corruption.
- **`ansi.StringWidth` returns 0 for tab characters — expand tabs before width math.** The `charmbracelet/x/ansi.StringWidth("\t")` function returns 0, but terminals render tabs as 1-8 characters (to the next tab stop). Any layout that computes padding, truncation, or column positions from `ansi.StringWidth` will miscalculate if the input contains tabs. The fix: call `expandTabs()` (in `diffparse.go`) to convert tabs to spaces early in the pipeline — before any width calculation, `ansi.Truncate`, or padding math. This applies to diff content (both split and unified views) and potentially any future panel that renders user-provided text. The same issue affects `lipgloss.Width()` and `runewidth.StringWidth()`.
- **Chroma's terminal formatter resets after every token — use `injectBg` for diff backgrounds.** The `writeToken()` function in chroma emits `\033[0m` (full SGR reset) after each syntax token. This means wrapping highlighted text with `lipgloss.Style.Render()` to set a background color only works for the first token — subsequent tokens lose the background. There is no chroma option to preserve an outer background. The fix is `injectBg(s, bgEsc)` which re-applies the background escape after every `\033[0m` in the string. This is used in both `formatSideContent` (side-by-side diff) and `RenderUnifiedLines` (unified diff) for added/removed line backgrounds.
- **`onFinish` must fire before session removal.** In the runner's exit goroutine, the `onFinish` callback must execute while the session is still in `r.sessions` so that consumers (e.g., daemon exit info cache) are populated before any observer sees the session as gone. The callback runs OUTSIDE `r.mu` to avoid deadlocking if it re-enters the runner (e.g., `HasSession`). The `delete(r.sessions, task.ID)` happens in a second lock section after the callback returns. This ordering prevents a race where `GetExitInfo` returns empty data because the session was deleted before exit info was cached.
- **`startOrAttach` must revert all state on Start failure.** When `runner.Start()` fails, revert `Status` to `Pending`, clear `SessionID`, and zero `StartedAt` before calling `db.Update`. Leaving the task in `InProgress` with no running session creates a ghost task that appears active but has no agent. The `SetStatus(StatusPending)` call doesn't clear `StartedAt` (only `InProgress` and `Complete` have timestamp side effects), so the explicit `t.StartedAt = time.Time{}` is required.
- **All daemon RPC calls must have a timeout.** Go's `net/rpc.Client.Call()` blocks indefinitely with no timeout. If the daemon crashes, every `refreshTasks()` tick (which calls `Running()` + `Idle()` via RPC) hangs the TUI. Use the `c.call()` wrapper (goroutine + select + `time.After`) for all RPC calls. The goroutine may outlive the timeout but will unblock when the OS delivers a connection error on the dead socket. The buffered channel (`make(chan error, 1)`) is critical — without it, the goroutine would block forever on send if the caller already timed out.

- **`IsSandboxAvailable()` is cached via `sync.Once` — safe to call unconditionally.** Checks `os.Stat("/usr/bin/sandbox-exec")` then `exec.LookPath("sandbox-exec")` — effectively free (a single syscall). All subsequent calls return the cached result. The settings view calls it on every refresh to show accurate status. `ResetSandboxCache()` clears the cache (for testing only).
- **sandbox-exec uses SBPL profiles, not JSON config.** `GenerateSandboxConfig()` writes a `.sb` temp file and returns `(profilePath, params []string, cleanup, err)` — a 4-return signature. `params` holds `["HOME=/...", "WORKTREE=/..."]` passed as `-D` flags. `WrapWithSandbox(cmdStr, profilePath, params)` produces `sandbox-exec -D 'HOME=...' -D 'WORKTREE=...' -f profile.sb sh -c 'original cmd'`. The double-sh is intentional (sandbox-exec wraps the shell, which wraps the agent command).
- **SBPL `(allow file-read*)` overrides subsequent `(deny file-read* (subpath X))` ONLY when X is a symlink-resolved path mismatch.** On macOS, `/tmp` is a symlink to `/private/tmp`. A deny rule on `(subpath "/tmp/foo")` does NOT block access to `/private/tmp/foo` because the kernel resolves symlinks before the rule is checked. Always use real, non-symlinked paths in deny rules (e.g., `(string-append (param "HOME") "/.ssh")` where HOME expands to `/Users/username` — a real path). The credential dir protect in the argus profile IS effective because HOME-relative paths are real paths. Never test SBPL rules using `/tmp` paths — use paths under `$HOME` instead.
- **sandbox-exec has no domain-level network filtering.** Unlike srt (which used a proxy to enforce domain allowlists), sandbox-exec's `network*` operations work at the socket/address level. Argus uses `(allow network*)` — all outbound connections are permitted. The sandbox protects FS writes and credential reads, not network egress. This is intentional and documented; do not attempt to add domain filtering via sandbox-exec profiles.
- **The sandbox profile must allow writes to `~/.claude.json` and `~/.claude/`.** Claude Code writes `~/.claude.json` (auth/session state) on every startup. Without this write permission, Claude emits ~41 bytes of terminal init sequences then hangs silently — the agent view appears blank with no output. Also allow `~/.claude/` (subpath) for conversation history writes needed by `--resume`. Blocking either write causes a silent hang, not an error message. The SBPL rules are: `(allow file-write* (literal (string-append (param "HOME") "/.claude.json")))` and `(allow file-write* (subpath (string-append (param "HOME") "/.claude")))`.
- **Symlink-resolved paths bypass subpath write rules.** If `~/.claude/skills` is symlinked to `~/.dots`, the `(allow file-write* (subpath "~/.claude"))` rule does NOT cover writes to `~/.dots/...` — the kernel resolves symlinks before checking. Reads are fine (global `(allow file-read*)` applies to the resolved path). Only add `~/.dots` to the write allow list if agents actually need to write there.
- **The sandbox profile must allow writes to the main repo's `.git` dir for worktree git operations.** Git worktrees store metadata (index, objects, refs) in the main repo's `.git/worktrees/<name>/` directory, not in the worktree itself. Without write access to `<main-repo>/.git`, `git add`, `git commit`, and `git push` all fail with `Operation not permitted`. `resolveGitDir()` in `sandbox.go` reads the worktree's `.git` file (which contains `gitdir: <path>`), walks up two levels from the gitdir path to find the `.git` directory, and adds it to the SBPL write allowlist. This happens automatically in `GenerateSandboxConfig()` — no user configuration needed.
- **Rebuilding the binary does NOT update the running daemon — restart required.** The daemon process loads the binary image at startup and runs it in memory indefinitely. Recompiling `go build -o argus ./cmd/argus/` updates the binary on disk but leaves the daemon process unchanged. Any new tasks started after the rebuild still use the OLD in-memory code (including the OLD `sandboxProfileBase` constant). To pick up sandbox profile changes (or any other daemon-side code changes), the daemon must be restarted: `kill -TERM $(cat ~/.argus/daemon.pid)` — the TUI will auto-restart it via `autoStartDaemon()`. Diagnose staleness by comparing `ls -la /path/to/argus` mtime against `ps -p $(cat ~/.argus/daemon.pid) -o lstart`.
- **Verify the sandbox profile in effect by reading the temp `.sb` file.** The daemon logs the profile path in `~/.argus/daemon.log` as `-f '/var/folders/.../T/argus-sandbox-NNNN.sb'`. While the sandboxed process is still running, `cat <path>` shows the exact SBPL profile being enforced. Compare it against `sandboxProfileBase` in `internal/agent/sandbox.go` to confirm the daemon is using current code. If the profile is missing `~/.claude.json`/`~/.claude/` write rules, the daemon is stale.
- **`sandbox.extra_write` DB entries produce literal SBPL `subpath` rules — garbage input produces broken rules.** The `sandbox.extra_write` config field is CSV-parsed and each value is wrapped in `(allow file-write* (subpath "..."))`. A partial UI entry (e.g., just `"e"`) produces `(allow file-write* (subpath "e"))` — syntactically valid but logically useless. To clear: `sqlite3 ~/.argus/data.sql "UPDATE config SET value='' WHERE key='sandbox.extra_write'"`.
- **Double-pointer fields must be allocated in `NewModel`, not in a post-construction setter.** `tea.NewProgram(m, ...)` copies `m` by value. Any `**T` field (like `program **tea.Program`) must have its outer pointer allocated in `NewModel` (`program: new(*tea.Program)`) so all BT value copies share the same inner slot. Calling `SetProgram(p)` after `NewProgram` only modifies the local `m`, not the copy inside the program. The pattern: allocate the `**T` in the constructor, then write through `*m.field = value` later — all copies see the write because they share the outer pointer. The same pattern applies to `restartedClient **dclient.Client`.
- **Daemon restart must preserve `SessionID` on tasks.** Claude Code's `--session-id` persists conversation state to disk. When the daemon restarts and kills all PTY processes, the `DaemonRestartedMsg` handler resets tasks to `Pending` but must NOT clear `SessionID`. On re-launch, `startOrAttach` sees `SessionID != ""`, sets `resume = true`, and passes `--resume <id>` to pick up the conversation where it left off. Clearing `SessionID` would lose the conversation history and start a fresh session.

## Development Rules

### Documentation Requirements

- **Every new feature must be documented in `context/knowledge/key-learnings.md` before the session ends.** Add a bullet to the Key Learnings section covering: what the feature does, where the code lives, any non-obvious design decisions, and edge cases handled. If the feature introduces a new pattern (e.g. a new message type, a new DB column, a new key binding), document the pattern so future agents don't have to rediscover it. (Key learnings are imported into CLAUDE.md via `@context/knowledge/key-learnings.md` — do NOT add new learnings directly to CLAUDE.md.)
- **Update `context/knowledge/code-quality.md`** with a dated section summarizing the feature's data model, flow, and any gotchas. Update `context/knowledge/index.md` to include the new key entities.
- **What to document:** new Task fields, new DB columns, new key bindings, new message types, new tea.Cmd patterns, new scan/detection loops, and any edge cases that required a second fix.

### Logging Requirements

- **Every new feature must include uxlog calls for debugging.** All async message handlers (tea.Msg case branches) that process results from external systems (GitHub API, git commands, daemon RPC, etc.) must log both success and failure paths via `uxlog.Log("[feature] ...")`. Use a consistent prefix per feature area (e.g., `[reviews]`, `[git]`, `[daemon]`). Without logs, failures are invisible — the UI just shows blank/stale data with no indication of what went wrong.
- **What to log:** fetch results (count/size), errors, state transitions, and any guards that silently skip work (e.g., cooldown timers, staleness checks).

### Testing Requirements

- **Every change must include tests.** When adding new functionality, fixing bugs, or refactoring, always add or update corresponding tests. Run `go test ./...` to verify all tests pass before considering work complete.
- **Run `go test ./... -cover` after writing tests** to verify coverage improved. Aim for ≥80% coverage on any package you touch.
- **Test file placement:** Tests go in `*_test.go` files in the same package (not `_test` suffix packages). Use the existing `testModel(t)`, `testDB(t)`, and similar helpers.
- **What to test:**
  - All exported functions and methods
  - Pure logic functions (parsers, state transitions, builders) — these are easy to test, no excuses
  - View/render functions — verify they return non-empty output and contain expected content strings
  - Edge cases: nil inputs, empty collections, boundary values, error paths
  - State machines: status transitions, cursor navigation, focus cycling
  - **Zero-dimension View() invariant:** When adding a new view or modifying a `View()` code path, add a subtest to `TestModel_ViewZeroDimensions` in `root_test.go`. This test calls `View()` with `width=0, height=0` (before `WindowSizeMsg`) and must not panic.
- **What's okay to skip:** Functions requiring a real terminal (raw mode, ioctl), functions that shell out to external processes (git commands, process management), and `cmd/argus/main.go` (entry point)
- **Testing patterns in this codebase:**
  - `db.OpenInMemory()` for database tests (no filesystem needed)
  - `agent.NewRunner(nil)` for UI tests that need a runner but don't start processes
  - `exec.Command("echo", "hello")` or `exec.Command("sleep", "10")` for agent/session tests
  - `DefaultTheme()` for any UI component tests
  - Table-driven tests with `[]struct{ input, expected }` for functions with many cases
  - **Keep daemon client test names short** — macOS Unix socket paths have a 104-byte limit. `t.TempDir()` + long test name exceeds it → `connect: invalid argument`. Use short names like `TestAlive_Dead` not `TestIsSessionAlive_DeadSession`.

## Context Directory

- `context/` stores durable cross-session knowledge checked into git
- `context/knowledge/index.md` is the knowledge graph index — read it when you need project history or domain context
- `context/research/` holds investigation notes and spike results
- `context/plans/` holds strategic plans and proposals

- **`git worktree add` requires a valid local ref or remote-tracking ref as the start point.** If the project config stores just `"master"` but the repo only has `origin/master` (no local branch), `git worktree add -b argus/task <path> master` fails with `fatal: not a valid object name: 'master'`. `resolveStartPoint()` in `worktree.go` resolves this by checking `git rev-parse --verify` and falling back to `upstream/<branch>` then `origin/<branch>`. The new project form also auto-detects the remote default branch (preferring `upstream` over `origin`) when the user enters a repo path, so new projects get the full ref like `upstream/master` by default.

- **Daemon cleanup must run on the `Serve` goroutine, not the `Shutdown` goroutine.** `Shutdown()` is always called from a non-main goroutine (signal handler or RPC handler). After `Shutdown` closes `d.done` and the listener, `Serve()` returns on the main goroutine → `main()` returns → process exits. If `StopAll()` and `removeIfOwnedByPID()` run inside `Shutdown()`, the process may exit before they complete, leaving zombie agent processes and stale socket/PID files. The fix: `Shutdown()` only signals (closes `d.done` + listener); `Serve()`'s accept-loop exit path calls `d.cleanup()` which does `StopAll` + file removal. Additionally, `signal.Stop(sigCh)` must be called after the signal handler goroutine exits — otherwise Go's `signal.Notify` captures SIGTERM into the buffered `sigCh` channel that nobody reads, silently swallowing the signal. This prevents `killExistingDaemon`'s SIGTERM from working, forcing a 2-second wait + SIGKILL escalation every time.

- **The sandbox profile must allow writes to `/var/folders`.** macOS uses `/var/folders` for per-user temp and cache directories (Go's `os.TempDir()`, `NSTemporaryDirectory()`, Node.js temp files, etc.). Without this write permission, agents that write to system temp dirs silently fail or crash. Unlike `/tmp` (a symlink to `/private/tmp`), `/var/folders` is a real path — the rule `(allow file-write* (subpath "/var/folders"))` works as-is with no symlink resolution needed.

- **Agent view file explorer must merge committed + uncommitted changes, not choose one or the other.** The original `UpdateGitStatus` used an if/else: show uncommitted files (`git status --short`) if any, *else* show committed branch files (`git diff --name-status base..HEAD`). This caused blank panels whenever the working tree was momentarily clean between agent commits — the committed changes existed but weren't shown because the uncommitted path returned nothing and the committed path was never reached. Fix: always call `MergeChangedFiles(ParseGitDiffNameStatus(msg.BranchFiles), ParseGitStatus(msg.Status))` which starts with committed branch files as the base and overlays uncommitted status (which takes precedence for the same path). This way the panel always shows all changes on the branch regardless of commit state.

- **File explorer directories auto-expand on cursor movement, one at a time.** `CursorUp()`/`CursorDown()` on `FileExplorer` call `autoExpand()` which expands the directory the cursor lands on (or its parent if on a child row) and collapses all others — same pattern as the task list's one-expanded-at-a-time behavior. `autoExpand()` saves `cursorPath` before rebuilding rows and restores the cursor to the same logical row afterward (critical because collapsing a directory shifts row indices). Returns the dir path needing a child fetch (empty string if cached or no dir). The agent view's key handler checks this return value and issues `fetchDirChildren()` as a `tea.Cmd` when needed. Enter on a directory opens the diff view (same as files) — no manual toggle. `ToggleDir()` still exists on `FileExplorer` for direct manipulation but is not used by the agent view.

- **Per-project sandbox config is stored as three columns on the `projects` table.** `sandbox_enabled` stores `""` (inherit global), `"true"`, or `"false"`. `sandbox_deny_read` and `sandbox_extra_write` store CSV paths appended to the global lists. `ResolveSandboxConfig(task, cfg)` merges global + per-project overrides before calling `GenerateSandboxConfig`. Adding new columns to an existing SQLite table uses `ALTER TABLE projects ADD COLUMN ... DEFAULT ''` — this is called after `CREATE TABLE IF NOT EXISTS` and the error for duplicate column names is silently ignored (covers both new and existing databases). `GenerateSandboxConfig` takes `config.SandboxConfig` directly (not `config.Config`) so the caller controls which merged config is used.

- **`textarea.LineInfo().ColumnOffset` is relative to the current visual row, not the hard line.** When a single hard line soft-wraps to multiple visual rows, `ColumnOffset` gives the rune offset from the start of the *current visual row*, not from the start of the hard line. Use `li.StartColumn + li.ColumnOffset` to get the true position within the hard line (`m.col`). `StartColumn` is the cumulative rune count of all visual rows before the current one. Getting this wrong makes word navigation (Option+Left/Delete) think the cursor is near position 0 when it's on a wrapped row, causing it to jump to the first visual line. The existing multi-line tests didn't catch this because they used short hard lines that never wrapped — always test with narrow textarea widths to exercise soft-wrap paths.

- **PR URL detection scans session output on each tick, not just on finish.** The `TickMsg` handler calls `runner.Running()` and for each running session calls `prURLRe.FindAllString(string(sess.RecentOutputTail(32*1024)), -1)`, scanning only the last 32KB to avoid copying the entire ring buffer every tick. Takes the last match so the most recently opened PR wins. The guard `t.PRURL != url` prevents redundant DB writes when the URL hasn't changed.

- **Also scan `msg.LastOutput` in `handleAgentFinished` to catch fast-finishing agents.** `runner.Running()` no longer includes a task once its session exits. If an agent creates a PR and exits before the next 1-second tick, the tick scan misses the URL. Fix: scan `msg.LastOutput` inside `handleAgentFinished` when `t.PRURL == ""`. This is the only place in the exit path that still has access to the ring buffer contents.

- **The ring buffer must be bounded (256KB), not unbounded.** Setting `defaultBufSize = 0` (unbounded, grow-only) causes CPU spikes and OOM for long-running sessions because `RecentOutput()` copies the entire buffer, and it's called on every tick from multiple hot paths (PR URL scan, agent view render, preview panel). The session log file (`~/.argus/sessions/<taskID>.log`) persists full PTY output to disk for post-restart scrollback, so the in-memory buffer only needs to hold enough for incremental vt10x feeding and live scrollback. Use `RecentOutputTail(n)` / `RingBuffer.Tail(n)` when only a small tail is needed (e.g., PR URL scanning only needs last 32KB). The `renderIncremental` path's wrap detection (`newBytes > uint64(len(raw))`) correctly handles the bounded buffer — it triggers a full vt10x reset+replay of the 256KB buffer when the ring wraps.

- **Adding a new column to an existing SQLite table uses `ALTER TABLE ... ADD COLUMN ... DEFAULT ''` after `CREATE TABLE IF NOT EXISTS`.** The error for a duplicate column (column already exists) is silently ignored with `//nolint:errcheck`. This pattern covers both fresh databases (CREATE TABLE includes the column) and existing ones (ALTER TABLE adds it). Used for `pr_url` on tasks and `sandbox_enabled/deny_read/extra_write` on projects. Any future column additions must follow this same pattern in `createTables()` — never use schema migrations for simple column additions.

- **`taskColumns` is the canonical column list used in every task query.** When adding a new column to the tasks table, update `taskColumns`, `scanTask` (add `&t.NewField` in the exact same position), `Add` (insert/values), and `Update` (SET clause) all in lockstep. Missing any one of these causes silent data loss or scan errors that only manifest at runtime.

- **Use `prURLRe.FindAllString(output, -1)` and take `matches[len(matches)-1]` for PR URL detection.** Taking the last match means the most recently opened PR wins — correct when an agent opens a draft, revises it, and opens a final PR. The regex `https://github\.com/[a-zA-Z0-9_.\-]+/[a-zA-Z0-9_.\-]+/pull/\d+` covers both plain-text URLs and OSC 8 hyperlinks (where the URL appears twice — as target and display text). The final match is the display text occurrence.

- **`'o'` key to open PR in browser from task list uses `exec.Command("open", url).Start()` inside a `tea.Cmd`.** The `open` command on macOS opens any URL in the default browser. The cmd must return a `tea.Msg` (return nil) — not just `exec.Command(...).Start()` directly in Update — because starting a process in Update would block. Add the key case using `msg.String() == "o"` (consistent with agent view's inline `case "o":` switch pattern) rather than adding it to KeyMap, since it is not user-configurable.

- **`gh search prs --json` does NOT support `reviewDecision` — use `gh pr list` instead.** The GitHub Search API returns `SearchResultItem`, not `PullRequest`. Available search fields: `number`, `title`, `author`, `isDraft`, `repository`, `updatedAt`, `url`, `state`, `labels`, `body`, `commentsCount`, `createdAt`, `closedAt`, `id`, `isLocked`, `isPullRequest`, `authorAssociation`, `assignees`. `reviewDecision` is available via `gh pr list --json` and `gh pr view --json` (both use the GraphQL PullRequest type). `FetchPRList()` in `internal/github/github.go` fetches the cross-repo list via `gh search prs`, then enriches with `reviewDecision` via `gh pr list` grouped by repo — O(repos) not O(PRs).

- **Reviews tab: `internal/ui/reviews.go` + `internal/github/github.go`.** Three-panel layout (PR list / diff / comments) using `PanelLayout`. PR list fetched via `gh search prs`, files via REST API, full diff via `gh pr diff` (cached per PR, re-sliced per file via `ExtractFileDiff`). Comments via REST. Review actions: `a` approve, `r` request changes, `c` line comment. Focus cycles: list → diff → comment compose. Auto-refresh on tick: comments have 2min TTL, diff staleness checked against `PR.UpdatedAt`. All message handlers log to uxlog with `[reviews]` prefix. **`SetPRs` must sort review requests before "my PRs" in the slice** — `renderPRList()` renders "Review Requests" first and "My Open PRs" second, but `prCursor` navigates the flat slice sequentially. If the slice order doesn't match the visual order, the cursor starts in the wrong section and review requests appear unreachable. **PR list caching:** `prListCooldown` (10min) gates refetch on tab entry — all three tab-switch paths (`"2"`, TabRight, TabLeft) check `canFetchPRList()`. `SetPRs` preserves cursor, scroll offset, and selection on background refreshes (only resets on first load). `View()` shows cached data during background refresh with a dimmed "refreshing…" indicator; errors show as "refresh failed: ..." rather than replacing the list. `'R'` forces a manual refresh subject to the same cooldown.

- **Task rename is display-only — worktree dir and branch are unchanged.** The 'r' key in the task list opens `viewRenameTask` with a `RenameTaskForm` (single `textinput.Model`, in `internal/ui/renametask.go`). Rename only updates `t.Name` in the DB — the worktree directory path, git branch, and session ID are all preserved. This means rename works even while an agent is running.

- **Archive section uses `Task.Archived` bool persisted as INTEGER in SQLite.** Press `'a'` on any task in the task list to toggle it in/out of the archive. Archived tasks are removed from the main project groups and displayed in an auto-expanding "Archive" section at the bottom of the task list. The archive auto-expands when the cursor enters it (lands on or past the archive header) and auto-collapses when the cursor leaves (moves above the archive header). There is no manual toggle — `autoExpand()` in `tasklist.go` handles both archive section expansion and per-project expansion within the archive. The archive section has its own project sub-groups with independent expansion (`archiveProject` field on `TaskList`, separate from `expanded`). The archive header is a navigable row (`rowArchiveHeader`) — cursor can land on it. `buildRows()` separates tasks by `t.Archived` before grouping by project — projects with only archived tasks never appear in the main section. `isInArchiveSection(idx)` walks backward from a row index to find if it's after the archive header — used by `autoExpand()` and `renderProjectHeader()` to use the correct expansion state. The `archived` DB column uses the standard `ALTER TABLE ... ADD COLUMN ... DEFAULT 0` migration pattern.

- **Daemon process appears as "argusd" in Activity Monitor via symlink.** `AutoStart` in `internal/daemon/client/client.go` creates a symlink `~/.argus/argusd -> <real binary>` and launches the daemon via that symlink. macOS Activity Monitor shows the basename of the executable path as the process name. The symlink is updated whenever `os.Executable()` differs from the current target (handles binary rebuilds). Falls back to the real binary path if symlink creation fails. The symlink persists on disk (never cleaned up) but is harmless when stale — `AutoStart` recreates it on next launch.

- **Codex resume is encoded in argus, not user-configurable.** `Backend.ResumeCommand` was removed. Instead, `IsCodexBackend(command string) bool` detects codex by `filepath.Base(firstWord) == "codex"` (handles absolute paths like `/usr/local/bin/codex`). For codex, `BuildCmd` uses `codexResumeCmd + " " + shellQuote(task.SessionID)` on resume. The `resume_command` DB column is kept for backward compat but never read/written.

- **`codex resume --last` is unreliable for multi-session argus.** `--last` picks the globally most recent codex session regardless of directory — it bypasses the picker (which would normally filter by cwd). With multiple worktrees running codex, `--last` in worktree A resumes worktree B if it was more recently active. The correct approach: `codex resume --dangerously-bypass-approvals-and-sandbox <session-id>` with the specific UUID from the task.

- **Codex session ID is captured post-exit from `~/.codex/state_5.sqlite`.** After a codex session exits cleanly, `handleAgentFinished` fires a `tea.Cmd` that queries `SELECT id FROM threads WHERE cwd = ? ORDER BY updated_at DESC LIMIT 1` against codex's state DB. The result is returned as `CodexSessionCapturedMsg` and stored in `task.SessionID`. On subsequent runs, `resume = t.SessionID != ""` → `codex resume --dangerously-bypass-approvals-and-sandbox <id>`. The `_5` suffix in `state_5.sqlite` is codex's schema version (constant `codexStateDB`); update if codex migrates. Session IDs are validated as UUIDs before use in command construction.

- **Backend form: `internal/ui/backendform.go`.** Mirrors `ProjectForm` pattern with 3 textinput fields: Name, Command, Prompt Flag. Supports edit mode (name read-only) and new mode. Wired via `viewNewBackend`/`viewEditBackend` in root.go. Settings view keys: `[n]` new backend, `[e]` edit backend, `[d]` set as default (on backend rows). The BACKENDS section header shows `(default: <name>)`.

- **New task form has a backend selector.** Field order: project → backend → prompt. The backend selector shows sorted backend names only — no `(default)` entry. The configured default backend is pre-selected by name at open (`backendIdx` set by matching `defaultBackend` param, falls back to index 0). `SelectedBackend()` always returns the actual name at `backendIdx`; `Task().Backend` is always an explicit backend name (never empty string for "inherit").

- **`fixupBackends()` migrates old codex flags to `--dangerously-bypass-approvals-and-sandbox`.** Detection: `name == "codex"` AND command doesn't contain `--dangerously-bypass-approvals-and-sandbox`. Covers `--yolo` and `--full-auto`. Scoped to `name == "codex"` — users who renamed their codex backend must update manually. The `resume_command` DB column exists for backward compat but is no longer read or written.

- **`restoreCursor` must filter by archive section to avoid cross-section project name collisions.** When a project has both active and archived tasks, its name appears as a `rowProject` header in BOTH the main section and the archive section. The old `restoreCursor` iterated rows and took the first match on `(kind, project)` — always finding the main-section header, jumping the cursor out of the archive. Fix: `restoreCursor` now takes an `inArchive bool` and skips candidate rows where `tl.isInArchiveSection(i) != inArchive`. Both call sites in `autoExpand()` pass the correct value (true for archive, false for main).

- **KB (Knowledge Base) is a SQLite FTS5 full-text search store baked into the Argus daemon.** `internal/kb/` defines `Document`, `SearchResult`, and `KBStore` interface. `internal/db/kb.go` implements `KBStore` on `*db.DB` using FTS5 (`kb_documents` virtual table) and a companion `kb_metadata` table for non-text fields (timestamps, word count). FTS5 doesn't support UPDATE — the upsert pattern is DELETE+INSERT in a transaction. `kb_pending_tasks` table holds Obsidian-detected tasks awaiting human approval. The `kb` package does NOT import `internal/db`; only the other way around. The FTS5 table is created in `createTables()` in `db.go` with the porter unicode61 tokenizer.

- **MCP server (`internal/mcp/`) exposes the KB as a streamable HTTP JSON-RPC 2.0 endpoint.** Listens on `POST /mcp`. Implements four tools: `kb_search`, `kb_read`, `kb_list`, `kb_ingest`. Port default is 7742, auto-increments to 7750 on conflict. `New(db KBQuerier, port int)` takes an interface so it doesn't import `internal/db` directly. Key Codex bug workaround: echo back the client's `protocolVersion` in the `initialize` response (Codex v0.47 sends `2025-06-18` but requires it echoed back). `ListenAndServe()` starts the server in a goroutine and returns immediately with the actual port — callers use this port for config injection.

- **Auto-injection (`internal/inject/`) writes MCP config into Claude Code and Codex automatically.** `inject.InjectGlobal(port)` mutates only the `mcpServers.argus-kb` key in `~/.claude.json`, preserving all other keys. `inject.InjectWorktree(path, port)` writes `<worktree>/.mcp.json`. `inject.SetClaudeProjectMcpTrust()` sets `enableAllProjectMcpServers: true` in `~/.claude/settings.json` to suppress the first-use approval prompt. Codex injection is in the `inject/codex` sub-package (separate package to avoid name collision with the parent). Codex config uses targeted string manipulation to avoid a TOML library. The `inject.SetMCPPort(port)` / `inject.MCPPort()` pair uses `sync/atomic` to share the port with `agent/worktree.go` without locking. `inject.InjectWorktreeAll(path)` is called from `agent.CreateWorktree()` after the git worktree is created.

- **FTS5 virtual table creation uses `CREATE VIRTUAL TABLE IF NOT EXISTS ... USING fts5(...)` — not a regular CREATE TABLE.** The `USING fts5(...)` clause selects the FTS5 module. Columns marked `UNINDEXED` (path, tier) are stored but not indexed for full-text search. The `snippet()` function in FTS5 queries takes `(table, column_index, highlight_start, highlight_end, ellipsis, max_tokens)` — column indices are 0-based and count only the non-UNINDEXED columns differently depending on the FTS5 version. Use `ORDER BY rank` for BM25 ranking (lower rank = better match). The FTS5 `MATCH` operator requires sanitized queries — strip `" * ( ) : ^ { }` characters using `kb.SanitizeQuery()`.

- **Obsidian task syntax parsing (`internal/import/obsidian.go`) requires `#argus` to be mandatory in the regex.** Using `(?:\s+#argus(...)?)?$` (optional `#argus`) causes the regex to match ALL unchecked task lines — not just those tagged `#argus`. The correct pattern makes `\s+#argus` required: `^- \[ \] (.+?)\s+#argus(?:/([^\s]+))?(?:\s.*)?$`. The `(?:\s.*)?$` suffix allows trailing text after the project tag. Duplicate detection uses a `seen` map keyed by `sourceFile + "::" + taskName`.

- **FTS5 `SanitizeQuery` must strip all operator characters including `-` and `+`.** `-` is the FTS5 NOT operator and `+` is the proximity operator. Full set to strip: `" * ( ) : ^ { } - +`. Omitting `-` or `+` allows users (or agents via `kb_ingest`) to inject FTS5 query syntax, causing unexpected search results or errors. The canonical list is in `internal/kb/search.go`.

- **FTS5 + metadata JOIN avoids N+1 under mutex.** `KBSearch` must not issue a per-row `d.conn.QueryRow` inside the `rows.Next()` loop while `d.mu` is held. The nested `QueryRow` also needs `d.mu` → deadlock. Fix: use `LEFT JOIN kb_metadata km ON km.path = kb_documents.path` with `COALESCE(km.modified_at, 0)` in the main query. Same pattern as `KBList`. The `snippet()` FTS5 function still works when the FTS5 table is referenced by name (not alias) in the `snippet()` call even when the table is aliased in the FROM clause.

- **MCP `http.Server` must be stored on the struct for orderly shutdown.** `mcp.Server.ListenAndServe()` starts a goroutine and returns the actual port. Store the `*http.Server` as `s.httpSrv` and expose `Shutdown(ctx context.Context) error`. Daemon stores `mcpServer *mcp.Server` and calls `d.mcpServer.Shutdown(ctx)` in `cleanup()` with a 5-second context timeout. Without this, the MCP HTTP server goroutine leaks on daemon shutdown.

- **All config file writes should be atomic (temp + rename).** `os.WriteFile` is not atomic — a crash mid-write produces a truncated file. Use `os.CreateTemp(dir, pattern)` + write + close + `os.Rename` for any config file injection (Claude JSON, Codex TOML). The `inject/claude.go` already does this; `inject/codex/codex.go` was missing it and now follows the same pattern.

- **Never derive DB config keys from human-readable label strings.** `NewKBVaultForm(theme, vaultName, configKey, currentPath)` accepts the DB key explicitly. Previously derived from `vaultName[:5] == "Argus"` — fragile to label wording changes. The `configKey` is a stable DB identifier (`"kb.metis_vault_path"`, `"kb.argus_vault_path"`); the `vaultName` is display text. Keep them separate.

- **`filepath.Walk` must return error when the root itself is inaccessible.** The walk callback receives `(path, info, err)`. When `err != nil && path == vaultPath` (the root), return `err` — this propagates to `FullScan()` which propagates to `Indexer.Start()`. For sub-paths, `return nil` to skip gracefully. Without the root check, `FullScan` returns `nil` even when the vault directory doesn't exist, making misconfiguration silent.

- **KB Indexer must be started (and stopped) by the daemon.** `kb.NewIndexer(db, vaultPath).Start()` performs the initial full scan and launches the background watcher goroutine. Start it in `daemon.Serve()` after MCP is up, inside a goroutine so scan errors don't block `Serve`. Stop it in `daemon.cleanup()` before MCP shutdown. Store both `kbIndexer *kb.Indexer` and `mcpServer *mcp.Server` on the `Daemon` struct.

- **`ARGUS_UI_RUNTIME` must fail fast on unknown values and default to Bubble Tea.** Runtime selection happens before launching the TUI via `ParseRuntime()` in `internal/app/agentview/runtime.go`. Empty or unset values map to `bubbletea`; unknown values return a clear startup error. Input is normalized (lowercase, trim). Reserving `tcell` as accepted-but-not-yet-implemented (exits with code 2) makes the migration target explicit. The types live in `internal/app/agentview/` (not `internal/ui/`) so the future tcell runtime can import them without a circular dependency.

- **Center-panel PTY sizing rules live in shared helpers, not inline in each runtime.** `ptySizeForPanel(contentH, centerW)` in `internal/ui/terminal_adapter.go` computes `content height - 2` rows and `panel width - 4` cols, clamped to `5x20`, accounting for bordered panel chrome. `resizeSessionToPanel(sess, contentH, centerW)` and `forwardKeyToSession(sess, msg)` replace inline PTY math and key forwarding in `agentview.go`. Both use narrow unexported interfaces (`terminalResizeSession`, `terminalInputSession`) so any session type satisfying the methods works.

- **`internal/app/agentview/` is the runtime-agnostic agent view state layer.** This package defines `State` (scroll offset, focus, diff mode, task identity, git refresh timing), `Panel` (focus target enum), `DiffState`, `TerminalAdapter` (narrow interface a terminal pane needs: WriteInput, Resize, RecentOutput, TotalWritten, Alive, PTYSize), `SessionLookup` (resolves task ID → TerminalAdapter), and `UIRuntime` with `ParseRuntime()`. The existing `internal/ui/agentview.go` remains the Bubble Tea implementation; `internal/app/agentview` provides the contracts that a future tcell runtime would implement against. The `TerminalAdapter` interface is a strict subset of `agent.SessionHandle` — any SessionHandle satisfies it, so no adapter wrappers are needed.

- **`internal/tui2/` is the tcell/tview UI runtime.** Selected via `ARGUS_UI_RUNTIME=tcell` (Bubble Tea remains the default). The `App` struct owns the `tview.Application`, DB, runner, and all sub-views. Layout uses `tview.Flex` (vertical: header + pages + statusbar) with `tview.Pages` for view switching between task list, agent view, and new task form. Sub-views are custom `tview.Box`-based widgets (`Header`, `StatusBar`, `TaskListView`, `TerminalPane`, `GitPanel`, `FilePanel`, `NewTaskForm`) that implement `Draw(screen)` directly. Mouse enabled via `tapp.EnableMouse(true)`.

- **`TerminalPane` renders vt10x cells directly to tcell.Screen — no ANSI intermediary.** Both live mode (`renderLive` — incremental byte feeding) and scrollback/replay mode (`renderReplay` — full buffer through tall vt10x) paint cells directly to the tcell screen via the shared `paintVT()` method. No `activeInputBG`, `findInputRow`, or prompt-row highlighting — the native surface shows upstream PTY output as-is. `cellStyle()` maps vt10x attributes to `tcell.Style`. The cursor is rendered unconditionally with high-contrast colors (`PaletteColor(17)` fg / `PaletteColor(153)` bg). `estimateVTRows()` is reimplemented locally to avoid importing `internal/ui` (which depends on Bubble Tea).

- **`vtColorToTcell` maps vt10x colors to tcell colors with correct default handling.** `DefaultFG`/`DefaultBG` → `tcell.ColorDefault` (inherits terminal theme). Colors 0-255 → `tcell.PaletteColor(n)`. Colors > 255 → 24-bit RGB via `tcell.NewRGBColor()`.

- **Inline diff viewer in `TerminalPane`.** `EnterDiffMode(diff, fileName)` parses raw unified diff into `[]diffLine` with types (added/removed/context/header) and renders with colored text. `ExitDiffMode()`, `ToggleDiffSplit()`, `DiffScrollUp/Down` for navigation. Diff is fetched asynchronously via `ui.FetchFileDiff` and rendered in the center panel, replacing the terminal view.

- **Agent view key layering matches Bubble Tea: 3-level exit.** `ctrl+q` and `esc` use: (1) if in diff mode → exit diff, refocus terminal; (2) if focused on files panel → refocus terminal; (3) otherwise → exit agent view. `updateFocusIndicators()` syncs border styles.

- **Session log loading for finished tasks.** `SetTaskID()` automatically reads `~/.argus/sessions/<taskID>.log` when no live session exists. The replay data is rendered via `renderReplay` (direct vt10x→tcell, same as scrollback).

- **`NewTaskForm` in tui2 is a modal overlay via `tview.Pages.AddPage`.** The `modeNewTask` view mode intercepts all global keys and delegates to the form's `InputHandler`. On submit, it creates the worktree before persisting the task (same safety invariant as Bubble Tea). On cancel/submit, the page is removed via `RemovePage`.

- **`GitPanel` and `FilePanel` replace the generic `SidePanel` in agent view.** `GitPanel` receives git status/diff/branch data via `SetStatus()` and renders colored sections. `FilePanel` mirrors the Bubble Tea `FileExplorer` with auto-expand, cursor navigation, and status icons — uses `ui.ChangedFile` directly (no duplication of parse logic).

- **tcell has no `KeyCtrlLeft`/`KeyCtrlRight` — use modifier checks.** Panel switching checks `event.Modifiers()&(tcell.ModCtrl|tcell.ModAlt) != 0` on arrow keys. Both Ctrl+arrow and Alt+arrow (macOS Cmd+arrow) work.

- **Task list three-panel layout: tasks | (git status + preview) | details.** The task list page (`taskPage` in `app.go`) uses a `tview.FlexColumn` with 1:3:1 ratio (20/60/20). The center column is a vertical `FlexRow` split 3:7 — `taskGitPanel` (top, ~30%) showing git status/diff for the selected task's worktree, and `TaskPreviewPanel` (bottom, ~70%) showing a terminal snapshot. This matches the old Bubble Tea layout where `splitCenterHeights` allocated 30% to git status (clamped 5-15 rows) and 70% to preview. `taskGitPanel` is a separate `GitPanel` instance from the agent view's `gitPanel`. `TaskPreviewPanel` pre-renders a cell grid via `xvt.SafeEmulator` in `RefreshOutput()` (called from the tick goroutine); `Draw()` only paints cached cells — zero blocking. `Draw()` must never call `runner.Get()`, `sess.RecentOutput()`, `os.ReadFile()`, or create a VT emulator. `fetchTaskGitStatus` runs git status for the selected task and updates `taskGitPanel` via `QueueUpdateDraw`. `TaskDetailPanel` (`taskdetail.go`) shows task metadata. All panels update via `OnCursorChange` callback and periodic tick refresh.

- **Task names with git-invalid characters must be sanitized before branch/dir creation.** `sanitizeBranchName()` in `internal/agent/worktree.go` strips characters forbidden by `git-check-ref-format`: spaces, control chars, `~ ^ : ? * [ ] { } \`, leading/trailing dots, consecutive dots, and `@{`. The sanitized name is used for both the branch (`argus/<safeName>`) and the worktree directory (`~/.argus/worktrees/<project>/<safeName>`). The task's display name (`t.Name`) is set to the sanitized+suffixed `finalName` returned by `CreateWorktree`. Worktree removal uses stored `t.Worktree` paths, not re-derived names, so no mismatch risk. Without sanitization, characters like `?` cause `git worktree add` to fail with exit status 255.

- **Empty task list shows centered banner with hint.** `TaskPage` (`internal/tui2/taskpage.go`) wraps the 3-panel task list layout. When `tasklist.HasTasks()` is false, it draws the banner (reusing `drawBanner` from `banner.go`) centered vertically with "Press [n] to create your first task" below — same pattern as `SettingsPage`. When tasks exist, it delegates to the inner `tview.Flex` normally. `InputHandler` and `MouseHandler` always delegate to the inner flex.

- **Never call `GetInnerRect()` or any tview widget method from the tick goroutine.** tview is not thread-safe — widget state is set during `Draw()` on the main goroutine. Cross-goroutine reads cause data races. The fix for `SyncPTYSize`: `Draw()` computes the desired PTY size under `tp.mu` and stores it in `pendingResizeRows`/`pendingResizeCols`. The tick goroutine atomically reads and clears the pending values under the same lock, then performs the resize RPC. This keeps all tview access on the main goroutine and all RPC calls off it.

- **`refreshTasks()` must not do RPC while holding `a.mu`.** `a.runner.Running()` is an RPC call (up to 5s timeout). Calling it inside `a.mu.Lock()` blocks the entire UI — any `QueueUpdateDraw` callback that also needs `a.mu` is stalled, causing a visible freeze. Fix: fetch `runningIDs` OUTSIDE the lock (same pattern as `onTick`), then lock and call `refreshTasksWithIDs(runningIDs)`.

- **Daemon session exits must be wired to the tui2 app via `client.OnSessionExit`.** Without this callback, when an agent process exits in daemon mode, the TUI never learns about it — `task.Status` stays `InProgress` forever and the agent view shows stale state. The callback fires `HandleSessionExit(taskID, info)` which updates the task to `Complete` (or `Pending` if stopped) via `QueueUpdateDraw`. For `StreamLost` exits, status is preserved since the process may still be alive.

- **x/vt `SafeEmulator` hangs on terminal query sequences — use `newDrainedEmulator()`.** The `charmbracelet/x/vt` emulator uses `io.Pipe()` internally for terminal responses. When processing DA1 (`\x1b[c`), DA2 (`\x1b[>c`), DSR (`\x1b[6n`, `\x1b[5n`), and similar query sequences, the emulator writes responses to the pipe writer which blocks indefinitely because nobody reads the pipe reader. This causes `emu.Write()` to hang, blocking the tview main goroutine and freezing the entire UI. Claude Code and Codex both emit these sequences to detect terminal capabilities. The fix: `newDrainedEmulator(cols, rows)` in `terminalpane.go` creates the emulator and immediately starts `go io.Copy(io.Discard, emu)` to drain the response pipe. This handles ALL query sequences generically — no need to strip specific escape sequences from the input. Both `TerminalPane` (live + replay rendering) and `TaskPreviewPanel` (task list preview) use `newDrainedEmulator()`. The drain goroutine exits when the emulator is garbage collected. Never use bare `xvt.NewSafeEmulator()` in the tui2 package.

- **`ctrl+d` must exit agent view when session is dead.** When a Claude/Codex session finishes (Ctrl+D sends EOF to the agent), the PTY process exits and `sess.Alive()` returns false. But pressing Ctrl+D again in the agent view did nothing — `tcellKeyToBytes` converts it to `{0x04}` which reaches the "Forward to PTY" block, but with no alive session the key is silently dropped. Fix: in `handleAgentKey`, check for `KeyCtrlD` when `sess == nil || !sess.Alive()` and call `exitAgentView()`. This mirrors the existing `'o'` key guard pattern for dead sessions.
